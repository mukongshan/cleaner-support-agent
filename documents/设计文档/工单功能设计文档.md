# 工单功能设计文档

## 文档说明

本文档详细说明清洁机器人智能指南平台后端系统的工单管理功能设计，包括工单的创建、受理、分派、处理、反馈、归档等全流程管理，以及工单类型自定义、工单状态跟踪等扩展功能。

**版本：** v1.0  
**更新日期：** 2024-01-20  
**作者：** 开发团队

---

## 1. 功能概述

### 1.1 功能目标

工单管理系统旨在实现运维工单的全流程管理，确保运维服务需求得到及时响应与处理。系统支持：

- ✅ **工单创建**：用户可创建工单，描述问题或需求
- ✅ **工单受理**：系统自动受理或管理员手动受理工单
- ✅ **工单分派**：将工单分派给合适的工程师或处理人员
- ✅ **工单处理**：工程师处理工单，更新处理进度和结果
- ✅ **工单对话**：用户和工程师在工单中进行实时对话沟通（复用Conversation实体）
- ✅ **工单反馈**：用户对工单处理结果进行评价反馈
- ✅ **工单归档**：已完成工单自动归档，便于历史查询
- ✅ **工单类型自定义**：支持自定义工单类型，适应不同业务场景
- ✅ **状态跟踪**：实时跟踪工单状态变化，记录操作日志

### 1.2 核心价值

- **提升响应效率**：通过优先级和状态管理，确保紧急问题优先处理
- **流程标准化**：统一的工单处理流程，提高服务质量
- **可追溯性**：完整的状态变更记录，便于问题追溯和分析
- **用户满意度**：及时反馈机制，提升用户体验

## 2. 业务流程设计

### 2.1 工单生命周期

```
创建(created) → 受理(accepted) → 分派(assigned) → 处理中(processing) → 已完成(completed) → 归档(archived)
                                    ↓
                                已取消(cancelled)
```

### 2.2 详细流程说明

#### 2.2.1 工单创建流程

1. **用户创建工单**
   - 用户填写工单标题、描述、优先级
   - 可选择关联AI对话记录（relatedAiChatId），用于提供上下文信息
   - 可上传附件（图片、视频等）
   - 系统自动生成工单ID（格式：WO + 日期 + 序号，如 WO20240120001）
   - **系统自动创建工单对话**：创建成功后，系统自动创建一个Conversation（类型为TICKET_CONVERSATION），用于后续用户和工程师的对话沟通

2. **工单初始状态**
   - 状态：`pending`（待受理）
   - 创建时间：系统自动记录
   - 创建人：当前登录用户
   - 工单对话：已创建但尚未有消息

#### 2.2.2 工单受理流程

1. **自动受理**（可选）
   - 系统可根据工单类型和优先级自动受理
   - 适用于标准化的低优先级工单

2. **手动受理**（当前实现）
   - 管理员或客服人员查看待受理工单列表
   - 确认受理后，工单状态变更为 `processing`（处理中）
   - 可指定处理工程师（engineerName）
   - 可设置预计处理时间（estimatedTime）

#### 2.2.3 工单分派流程

1. **自动分派**（扩展功能）
   - 根据工单类型、优先级、工程师工作负载自动分派
   - 支持负载均衡算法

2. **手动分派**（当前实现）
   - 管理员根据工单内容选择合适的工程师
   - 设置工程师姓名（engineerName）
   - 设置预计处理时间（estimatedTime）
   - 工单状态保持为 `processing`

#### 2.2.4 工单处理流程

1. **工程师处理**
   - 工程师查看分配给自己的工单
   - 分析问题，制定解决方案
   - 在工单对话中与用户沟通（通过工单关联的Conversation实现）
   - 更新处理进度（通过comments字段记录）
   - 上传处理过程中的附件

2. **工单对话（独立于AI对话）**
   - 每个工单关联一个独立的Conversation（类型为TICKET_CONVERSATION）
   - 该对话与AI对话（AI_CONVERSATION）完全独立，互不关联
   - 用户和工程师可以在工单对话中发送消息
   - 消息角色：`user`（用户）、`engineer`（工程师）
   - 对话记录完整保存，便于追溯

3. **状态更新**
   - 处理过程中：状态保持为 `processing`
   - 处理完成：状态变更为 `completed`
   - 无法处理：状态变更为 `cancelled`，并说明原因

#### 2.2.5 工单反馈流程

1. **用户反馈**
   - 工单完成后，用户可对处理结果进行评价
   - 支持点赞（like）或点踩（dislike）
   - 可填写反馈意见（comment）
   - 反馈信息存储在 `TicketFeedback` 实体中

2. **反馈统计**
   - 统计工程师的满意度评分
   - 用于绩效考核和服务质量分析

#### 2.2.6 工单归档流程

1. **自动归档**
   - 工单状态为 `completed` 或 `cancelled` 超过30天自动归档
   - 归档后工单仍可查询，但不再出现在活跃工单列表中

2. **手动归档**
   - 管理员可手动将已完成工单归档
   - 归档后工单状态变更为 `archived`

## 3. 实体设计

### 3.1 Ticket（工单实体）

**表名：** `tickets`

**说明：** 存储工单的核心信息

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键（自增） |
| ticketId | String(50) | UNIQUE, NOT NULL | 业务ID，如 "WO20240120001" |
| user | User | FK, NOT NULL | 创建工单的用户（多对一关系） |
| title | String(200) | NOT NULL | 工单标题 |
| description | TEXT | NULL | 工单详细描述 |
| status | TicketStatus | NOT NULL | 工单状态（枚举） |
| priority | TicketPriority | NOT NULL | 优先级（枚举） |
| relatedAiChatId | String(100) | NULL | 关联的AI对话ID（创建工单时关联的AI咨询对话） |
| conversation | Conversation | FK, NULL | 工单对话（用户与工程师的对话，一对一关系） |
| attachmentUrls | List<String> | NULL | 附件URL列表（一对多关系） |
| engineerName | String(100) | NULL | 处理工程师姓名 |
| engineerId | String(50) | NULL | 处理工程师用户ID |
| estimatedTime | String(100) | NULL | 预计处理时间 |
| comments | TEXT | NULL | 处理备注/意见 |
| createdAt | LocalDateTime | NOT NULL | 创建时间（自动设置） |
| updatedAt | LocalDateTime | NOT NULL | 更新时间（自动更新） |

#### 重要说明

- **relatedAiChatId**：创建工单时，用户可选择关联之前与AI的咨询对话（类型为AI_CONVERSATION），仅用于提供上下文信息，与工单对话完全独立
- **conversation**：工单创建后，系统自动创建一个新的Conversation（类型为TICKET_CONVERSATION），专门用于用户和工程师的工单对话沟通，与AI对话无任何关联
- **engineerId**：工程师的用户ID，用于关联工程师账户，便于权限控制和消息推送

**注意：**
- `relatedAiChatId` 和 `conversation` 是两个完全独立的字段
- `relatedAiChatId` 是字符串类型，仅存储AI对话的ID作为参考
- `conversation` 是实体关联，指向工单专用的对话Conversation
- 工单对话和AI对话在数据层面完全隔离，互不影响

#### 状态枚举（TicketStatus）

| 值 | 说明 | 可转换状态 |
|----|------|-----------|
| pending | 待受理 | processing, cancelled |
| processing | 处理中 | completed, cancelled |
| completed | 已完成 | archived |
| cancelled | 已取消 | - |
| archived | 已归档 | - |

#### 优先级枚举（TicketPriority）

| 值 | 说明 | 处理时限建议 |
|----|------|------------|
| low | 低优先级 | 3-5个工作日 |
| medium | 中等优先级 | 1-2个工作日 |
| high | 高优先级 | 2-4小时内 |

#### 实体关系

```
User (用户)
  ├── 1:N → Conversation (会话)
  │       └── 1:N → Message (消息)
  │
  └── 1:N → Ticket (工单)
          ├── 1:1 → Conversation (工单对话)
          │       └── 1:N → Message (对话消息，role: user/engineer)
          └── 1:1 → TicketFeedback (工单反馈)
```

**关系说明：**

- **Ticket → Conversation（一对一）**：每个工单关联一个独立的Conversation（类型为TICKET_CONVERSATION），专门用于用户和工程师的工单对话
- **Conversation类型区分**：
  - `AI_CONVERSATION`：用户与AI助手的对话（原有功能），与工单无关
  - `TICKET_CONVERSATION`：用户与工程师的工单对话（新增功能），与AI对话完全独立
- **Message角色扩展**：
  - `user`：用户消息（两种对话类型都使用）
  - `assistant`：AI助手消息（仅在AI对话中使用）
  - `engineer`：工程师消息（仅在工单对话中使用）

**重要：**
- AI对话（AI_CONVERSATION）和工单对话（TICKET_CONVERSATION）是完全独立的两个Conversation实例
- 它们通过`type`字段区分，在数据层面完全隔离
- 工单的`relatedAiChatId`仅作为参考信息，不建立任何数据关联

### 3.2 Conversation扩展设计

**复用现有Conversation实体，增加类型区分**

#### Conversation实体扩展

需要在现有Conversation实体中增加字段：

```java
@Column(length = 50)
@Enumerated(EnumType.STRING)
private ConversationType type; // 对话类型

@OneToOne(mappedBy = "conversation")
private Ticket ticket; // 如果是工单对话，关联工单

public enum ConversationType {
    AI_CONVERSATION,      // AI对话（用户与AI助手）
    TICKET_CONVERSATION   // 工单对话（用户与工程师）
}
```

#### Message实体扩展

需要在现有Message实体中扩展MessageRole枚举：

```java
public enum MessageRole {
    user,        // 用户消息
    assistant,   // AI助手消息（AI对话中使用）
    engineer     // 工程师消息（工单对话中使用）
}
```

#### 设计优势

1. **代码复用**：复用现有的Conversation和Message实体，减少代码重复
2. **数据隔离**：通过`type`字段区分对话类型，AI对话和工单对话在数据层面完全隔离
3. **统一接口**：对话相关的API可以统一处理，只需区分对话类型
4. **数据一致性**：使用相同的数据结构，便于统一管理和查询
5. **扩展性好**：未来如需支持其他类型的对话，只需扩展枚举即可

#### 重要说明

- **数据隔离**：AI对话（AI_CONVERSATION）和工单对话（TICKET_CONVERSATION）是完全独立的Conversation实例
- **命名清晰**：
  - `relatedAiChatId`：关联AI对话的ID（仅作为参考，字符串类型）
  - `conversation`：工单对话的实体关联（工单专用，Conversation类型）
- **互不影响**：工单对话和AI对话在业务逻辑和数据存储上完全独立，互不干扰

### 3.3 TicketFeedback（工单反馈实体）

**表名：** `ticket_feedbacks`

**说明：** 存储用户对工单处理结果的反馈

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键 |
| ticket | Ticket | FK, UNIQUE, NOT NULL | 关联的工单（一对一关系） |
| type | FeedbackType | NOT NULL | 反馈类型：like(赞) 或 dislike(踩) |
| comment | TEXT | NULL | 反馈意见 |
| createdAt | LocalDateTime | NOT NULL | 创建时间（自动设置） |

#### 反馈类型枚举（FeedbackType）

| 值 | 说明 |
|----|------|
| like | 满意/点赞 |
| dislike | 不满意/点踩 |

## 4. 接口设计

### 4.1 接口概览

**基础路径：** `/api/cleaner-support/v2/tickets`

**认证要求：** 所有接口需要登录认证（Bearer Token）

### 4.2 创建工单

**接口路径：** `POST /tickets`

**功能说明：** 用户创建新的工单

**请求参数：**

```json
{
  "title": "传感器持续报错",
  "description": "已清理但无效，需报修",
  "priority": "medium",
  "relatedAiChatId": "conv_123",
  "attachmentUrls": [
    "https://cdn.example.com/image1.jpg",
    "https://cdn.example.com/video1.mp4"
  ]
}
```

**参数说明：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| title | String | 是 | 工单标题（1-200字符） |
| description | String | 否 | 工单详细描述 |
| priority | String | 否 | 优先级：low/medium/high（默认medium） |
| relatedAiChatId | String | 否 | 关联的AI对话ID（创建工单时关联的AI咨询对话） |
| attachmentUrls | List<String> | 否 | 附件URL列表 |

**说明：**

- 工单创建成功后，系统会自动创建一个新的Conversation（类型为TICKET_CONVERSATION）
- 该Conversation专门用于用户和工程师的工单对话沟通，与AI对话完全独立
- `relatedAiChatId`用于关联用户之前与AI的咨询对话（类型为AI_CONVERSATION），仅作为参考信息，不建立数据关联
- 工单对话和AI对话在数据层面完全隔离，互不影响

**响应示例：**

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "ticketId": "WO20240120001",
    "title": "传感器持续报错",
    "status": "pending",
    "priority": "medium",
    "createdAt": "2024-01-20T10:00:00"
  }
}
```

**错误响应：**

```json
{
  "code": 400,
  "message": "工单标题不能为空",
  "data": null
}
```

### 4.5 更新工单状态

**接口路径：** `PUT /tickets/{ticketId}`

**功能说明：** 更新工单状态和备注信息

**路径参数：**

| 参数名 | 类型 | 说明 |
|--------|------|------|
| ticketId | String | 工单ID |

**请求参数：**

```json
{
  "status": "completed",
  "comments": "问题已解决，设备正常运行"
}
```

**参数说明：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| status | String | 否 | 工单状态：pending/processing/completed/cancelled |
| comments | String | 否 | 处理备注/意见 |

**响应示例：**

```json
{
  "code": 200,
  "message": "工单状态更新成功",
  "data": null
}
```

**状态转换规则：**

- `pending` → `processing`：受理工单
- `processing` → `completed`：完成处理
- `processing` → `cancelled`：取消工单
- `completed` → `archived`：归档工单（需管理员权限）

### 4.6 工单对话接口（扩展接口）

**接口路径：** `GET /tickets/{ticketId}/conversation`

**功能说明：** 获取工单关联的对话信息（工单专用对话，与AI对话无关）

**响应示例：**

```json
{
  "code": 200,
  "data": {
    "conversationId": "conv_ticket_001",
    "type": "TICKET_CONVERSATION",
    "messages": [
      {
        "role": "user",
        "content": "设备还是无法正常工作",
        "timestamp": "2024-01-20T10:30:00"
      },
      {
        "role": "engineer",
        "content": "收到，我这边安排上门检查",
        "timestamp": "2024-01-20T10:35:00"
      }
    ]
  }
}
```

**接口路径：** `POST /tickets/{ticketId}/conversation/messages`

**功能说明：** 在工单对话中发送消息（工单专用对话，与AI对话无关）

**请求参数：**

```json
{
  "content": "设备还是无法正常工作"
}
```

**说明：**

- 该接口操作的是工单专用的Conversation（类型为TICKET_CONVERSATION）
- 与AI对话（AI_CONVERSATION）完全独立，互不影响
- 用户发送消息时，role自动设置为`user`
- 工程师发送消息时，role自动设置为`engineer`
- 消息会保存到工单关联的Conversation中

## 5. 状态流转规则

### 5.1 状态转换图

```
                    ┌─────────┐
                    │ pending │ (待受理)
                    └────┬────┘
                         │
            ┌────────────┼────────────┐
            │            │            │
            ▼            ▼            ▼
    ┌───────────┐  ┌───────────┐  ┌───────────┐
    │processing │  │cancelled  │  │processing │
    │(处理中)   │  │(已取消)   │  │(处理中)   │
    └─────┬─────┘  └───────────┘  └─────┬─────┘
          │                              │
          │                              │
          └──────────┬───────────────────┘
                     │
                     ▼
              ┌───────────┐
              │completed  │ (已完成)
              └─────┬─────┘
                    │
                    ▼
              ┌───────────┐
```

## 9. 数据库设计

### 9.1 表结构

**tickets 表：**

```sql
CREATE TABLE tickets (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_id VARCHAR(50) UNIQUE NOT NULL,
    user_id BIGINT NOT NULL,
    title VARCHAR(200) NOT NULL,
    description TEXT,
    status VARCHAR(20) NOT NULL,
    priority VARCHAR(20) NOT NULL,
    related_ai_chat_id VARCHAR(100),
    conversation_id BIGINT,
    engineer_name VARCHAR(100),
    engineer_id VARCHAR(50),
    estimated_time VARCHAR(100),
    comments TEXT,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

CREATE INDEX idx_ticket_user ON tickets(user_id);
CREATE INDEX idx_ticket_status ON tickets(status);
CREATE INDEX idx_ticket_created ON tickets(created_at);
CREATE INDEX idx_ticket_conversation ON tickets(conversation_id);
```

**conversations 表扩展：**

```sql
ALTER TABLE conversations 
ADD COLUMN type VARCHAR(50) DEFAULT 'AI_CONVERSATION',
ADD COLUMN ticket_id BIGINT,
ADD FOREIGN KEY (ticket_id) REFERENCES tickets(id);

CREATE INDEX idx_conversation_type ON conversations(type);
CREATE INDEX idx_conversation_ticket ON conversations(ticket_id);
```

**字段说明：**

- `type`：对话类型，用于区分AI对话（AI_CONVERSATION）和工单对话（TICKET_CONVERSATION）
- `ticket_id`：仅当type为TICKET_CONVERSATION时才有值，关联对应的工单
- **数据隔离**：AI对话和工单对话通过type字段完全隔离，互不影响

**messages 表扩展：**

```sql
-- MessageRole枚举扩展：user, assistant, engineer
-- 无需修改表结构，只需在代码中扩展枚举即可
-- role字段说明：
--   - user: 用户消息（两种对话类型都使用）
--   - assistant: AI助手消息（仅在AI对话中使用）
--   - engineer: 工程师消息（仅在工单对话中使用）
```

**重要说明：**

- `tickets.related_ai_chat_id`：字符串类型，仅存储AI对话的ID作为参考，不建立外键关联
- `tickets.conversation_id`：外键关联，指向工单专用的Conversation（type为TICKET_CONVERSATION）
- 这两个字段完全独立，互不关联
- AI对话和工单对话在数据层面完全隔离

**ticket_attachments 表：**

```sql
CREATE TABLE ticket_attachments (
    ticket_id BIGINT NOT NULL,
    attachment_url VARCHAR(500) NOT NULL,
    FOREIGN KEY (ticket_id) REFERENCES tickets(id)
);
```

**ticket_feedbacks 表：**

```sql
CREATE TABLE ticket_feedbacks (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_id BIGINT UNIQUE NOT NULL,
    type VARCHAR(20) NOT NULL,
    comment TEXT,
    created_at DATETIME NOT NULL,
    FOREIGN KEY (ticket_id) REFERENCES tickets(id)
);
```

### 9.2 索引设计

- `tickets.user_id`：用户查询工单列表
- `tickets.status`：按状态筛选工单
- `tickets.created_at`：按时间排序
- `tickets.ticket_id`：唯一索引，快速查找
