# APP 形态向移动端 Web 化改造设计说明

## 一、文档目的与背景

### 1.1 背景

当前系统按**移动 APP** 的形态设计与实现：单列布局、底部 Tab 切换、无 URL 路由、状态全在内存，用户入口假设为「打开已安装的应用」。甲方要求改为**移动端可访问的网页**，即用户通过浏览器（手机或电脑）访问同一套能力，无需安装，通过链接即可使用与分享。

### 1.2 文档目的

本文档从**产品与交互设计**、**信息架构**、**前端技术**、**后端与部署**四个维度，说明由「APP 形态」转向「移动端可访问的 Web」时，在**设计上需要做出的改动**，作为改造实施的依据，不涉及对现有某篇设计文档的参考或替换。

---

## 二、定位与目标变化

| 维度 | APP 形态（当前） | 移动端 Web（目标） |
|------|------------------|--------------------|
| **入口** | 桌面图标、应用内推送 | 浏览器地址栏、扫码、短信/邮件/聊天中的链接 |
| **分发** | 应用商店、内部分发包 | 无需安装，打开即用 |
| **使用场景** | 以「打开 APP」为起点 | 可能从任意链接进入（某条知识、某工单、问答页） |
| **设备** | 以手机竖屏为主 | 手机竖屏/横屏、平板、桌面浏览器均可能 |
| **分享** | 难以分享到「具体某一页」 | 需要支持「分享当前页面链接」且他人打开即到该页 |

设计上的核心转变：从**「一个封闭的、以应用为边界的交互流」**变为**「以 URL 为粒度的、可被链接与收藏的 Web 应用」**。

## 三、信息架构与导航设计改动

### 3.1 引入「URL 驱动」与路由

**现状（APP 思路）**：所有页面通过 Tab 状态（如 `activeTab`）切换，无 URL 变化，刷新即回到默认 Tab，无法收藏或分享到「工单详情」「某条知识」等具体视图。

**改造要点**：

- **为每个主要视图分配唯一路径**，例如：
  - `/` 或 `/chat`：问答
  - `/knowledge`：知识库
  - `/knowledge/:id`：某条知识详情（若产品需要）
  - `/tickets`：工单列表
  - `/tickets/:id`：工单详情
  - `/profile`：个人中心
  - `/login`、`/register`：登录/注册
- **路由与当前 Tab/页面状态同步**：用户点击底部 Tab 或内部跳转时，同步更新 URL；用户通过链接或浏览器前进/后退进入时，正确展示对应页面并高亮对应 Tab。
- **刷新、收藏、分享**：刷新后根据当前 URL 恢复页面，分享或收藏得到的是「当前页」的链接，他人打开直接进入该页（在权限允许前提下）。

这样，信息架构从「应用内多 Tab」升级为「可寻址的页面 + 导航结构」，便于 Web 场景下的使用与传播。

### 3.2 底部导航在 Web 多端下的表现

**现状**：底部固定 4 Tab（问答、知识库、工单、个人中心），布局为 `max-w-md mx-auto`，偏手机竖屏单列。

**改造要点**：

- **手机竖屏**：保留底部 Tab 栏，可继续使用 `safe-area-bottom` 等适配刘海/横条。
- **大屏（平板、桌面）**：设计上需决定：
  - **方案 A**：仍保留底部 Tab，内容区保持最大宽度（如 max-w-2xl）居中，两侧留白；
  - **方案 B**：大屏改为顶栏 Tab 或侧边栏导航，与「网页」心智更一致。
- 无论哪种，**当前页与 URL 一致**，且**返回/关闭**行为与浏览器前进/后退或历史栈一致（见下节）。

### 3.3 返回、深度链接与历史栈

**现状**：工单详情等二级页通过「返回」回到工单列表，无 URL，无法从外部直接打开某条工单。

**改造要点**：

- **返回**：优先使用「浏览器后退」或路由的 `navigate(-1)`，使地址栏与用户预期一致；界面上的「返回」按钮与浏览器后退**行为一致**，即回到**上一历史记录**，而非固定回到某 Tab 或首页（详见第八章 8.1 节「路由回退逻辑」）。
- **深度链接**：支持通过 `/tickets/123` 等形式直接打开工单详情（需鉴权与 404 处理）。
- **登录拦截**：未登录用户访问需登录的 URL 时，可跳转 `/login`，并在登录成功后**回到原目标 URL**，而不是固定进入首页或问答页。

## 四、响应式与布局设计改动

### 4.1 从「单列移动壳」到「多端适配」

**现状**：整体 `max-w-md`（约 448px）居中，适合手机，大屏上仍是窄条。

**改造要点**：

- **断点策略**：明确手机 / 平板 / 桌面的断点（如 768px、1024px），对导航、内容宽度、字体与触控区域做统一规划。
- **内容区最大宽度**：大屏下可为阅读与操作设定更大 max-width（如 672px 或 896px），避免一行过长。
- **横屏**：考虑导航栏高度、键盘弹起时的可视区域（尤其是输入框、底部按钮），必要时用 `env(safe-area-inset-*)` 与 viewport 单位配合。

### 4.2 安全区域与全屏体验

**现状**：已有 `viewport-fit=cover` 与底部 `safe-area-bottom` 类。

**改造要点**：

- 全站统一使用安全区域（刘海、底部横条、横屏时的左右边），避免关键按钮或文字被遮挡。
- 若需「类 APP」全屏体验，可配合 PWA 的 `display: standalone` 或 `minimal-ui`，在 meta 与 manifest 中声明；纯浏览器访问时仍保证安全区域正确。

### 4.3 触控与点击区域

- 底部 Tab、主要按钮等可点击区域建议不小于 44×44 逻辑像素，符合移动端可访问性。
- 若后续支持桌面端，需兼顾鼠标悬停与键盘焦点样式，与触控不冲突。

## 五、访问方式、部署与安全

### 5.1 访问方式与入口

**改造要点**：

- **固定访问地址**：通过统一域名（如 `https://support.xxx.com`）访问，生产环境使用 HTTPS。
- **入口形式**：支持扫码打开、短信/邮件/微信中的链接、桌面浏览器输入 URL，所有入口最终落到同一套 SPA 路由。
- **与公众号/小程序等的关系**：若需在微信内使用，需明确是「微信内浏览器打开 H5」还是「小程序内嵌 H5」；两者在鉴权、分享、支付等设计上会有差异，需单独约定。

### 5.2 前端部署与资源

**现状**：开发环境 Vite dev server，构建产物为静态资源，API 通过环境变量 `VITE_API_BASE_URL` 配置。

**改造要点**：

- **生产 base path**：若前端不部署在根路径（如 `https://xxx.com/app/`），需在构建与路由中配置 `base`，避免资源 404。
- **SPA 回退**：部署到 Nginx 等时，对非静态资源请求统一回退到 `index.html`，由前端路由接管。
- **API 地址**：生产环境通过 `VITE_API_BASE_URL` 指向正式后端，避免硬编码 localhost；`index.html` 中的 preconnect 等也需随环境变化或去掉硬编码。

### 5.3 鉴权与跨域

**现状**：JWT 存 localStorage，后端 CORS 配置允许指定前端源。

**改造要点**：

- **CORS**：生产环境将前端实际域名加入 `allowed-origins`，避免仅允许 localhost。
- **Token 安全**：浏览器中 localStorage 可被 XSS 读取，需保证：不引入不可信脚本、关键接口可考虑 HttpOnly Cookie 等（若后端支持）；若继续用 Token，需在传输与存储上遵循现有安全实践。
- **未授权处理**：当前有「弹登录框」与「跳转 /login」两种分支；Web 下若用户通过链接进入某页再被要求登录，建议跳转 `/login?redirect=当前 URL`，登录成功后跳回，体验更符合 Web。

## 六、用户体验与浏览器特性

### 6.1 地址栏、刷新与前进/后退

- **地址栏**：始终反映当前页面路由，用户可复制链接、收藏。
- **刷新**：刷新后根据 URL 恢复对应页面与数据，不丢失「当前在看什么」。
- **前进/后退**：与路由历史一致，避免「返回」只关模态框而浏览器历史未退」的割裂感；若有全屏弹层（如登录），需明确是「压栈」还是「替换当前历史」，避免用户多次点后退才关掉弹层。

### 6.2 分享与传播

- **分享到具体页面**：用户在某工单详情、某条知识页时，分享应得到该页 URL，他人打开即到该页（含鉴权与权限控制）。
- **SEO**：若需被搜索引擎收录（如知识库文章），可后续考虑 SSR 或预渲染；若仅内部使用，可暂不优先。

### 6.3 标题与多页感知

- 每个路由对应一个 `<title>`（及可选的 meta description），便于多标签页区分与收藏显示；例如「工单 #123 - 售后支持」。

## 七、技术实现要点汇总

| 类别 | 改动要点 |
|------|----------|
| **前端路由** | 引入 React Router（或等价方案），定义与 Tab/页面一致的路由表；Tab 点击与路由双向同步；子页（工单详情等）独立路由。 |
| **前端构建** | 生产 base path、`VITE_API_BASE_URL` 等环境变量；SPA 回退由部署层配置。 |
| **布局与响应式** | 断点与多端布局策略；大屏下导航形式（底部保留或顶栏/侧栏）；安全区域全站统一。 |
| **后端** | CORS 增加生产前端域名；未授权时可返回 401 或重定向逻辑与前端约定（如 redirect 参数）。 |
| **登录与授权** | 支持「登录后回跳原 URL」；深度链接在未登录时的跳转与登录后回跳。 |

## 八、技术实现细则

本章节对开发实施中的**异常处理**、**兼容性与性能**、**安全边缘场景**及**部署与运维**做明确约定，供开发团队直接对照执行，减少沟通成本。

### 8.1 前端路由的回退策略与 404 处理

#### 404 页面设计

- **约定**：用户访问不存在的路径（如错误输入 `/tickts`、或 `/tickets/999999` 且该工单不存在）时，**统一展示自定义 404 页面**，而非静默重定向到首页，以便用户明确「当前地址无效」并可通过页面上的入口返回首页或工单列表。
- **实现要点**：
  - 路由表中定义通配路由（如 `path: '*'` 或 `path: '/*'`）渲染 404 组件。
  - 对依赖后端数据的路由（如 `/tickets/:id`），在请求详情接口返回 404 时，前端切换到 404 页面或同一 404 组件并展示「工单不存在」等文案；不自动跳转到首页，避免用户误以为链接有效。
- **例外**：若产品明确要求「无效 ID 直接回首页」，可单独约定；默认建议采用「显式 404 页 + 返回首页/列表按钮」。

#### 路由回退逻辑（返回按钮与历史栈深度）

- **默认行为**：页面内「返回」按钮与浏览器后退**一致**，即执行 `navigate(-1)`（或等价），回到**上一历史记录**，而非固定回到某 Tab 或首页。
- **典型场景**：首页 → 工单列表 → 工单详情 →（若有）编辑页。用户在编辑页点击「返回」应回到**工单详情**；再点「返回」回到工单列表；再点回到首页。这样与浏览器历史栈一致，避免「返回」逻辑与用户预期不符。
- **边界**：若历史栈中上一页是站外（如从搜索引擎点进详情），`navigate(-1)` 会离开本站，可提示「将离开当前站点」或改为跳转到上一级路径（如从 `/tickets/123` 回 `/tickets`），需在设计中二选一并统一。

### 8.2 浏览器兼容性与性能监控

#### 浏览器兼容性矩阵

- **建议支持的最低版本**（可根据甲方环境再收紧）：
  - Chrome 80+
  - Safari 14+（含 iOS 14+）
  - Firefox 80+
  - Edge 80+（Chromium）
- **iOS 特别说明**：`viewport-fit=cover`、`env(safe-area-inset-*)` 在 iOS 11.2+ 支持，但在不同机型（刘海屏、灵动岛）上表现需在真机验证；建议在项目内维护「兼容性检查清单」，上线前在目标机型上回归关键页面。
- **构建目标**：前端通过 `browserslist` 或 Vite 的 `build.target` 与上述范围对齐，避免产出过新语法导致旧版浏览器报错。

#### 性能监控与指标

- **Web Vitals**：对以下指标做采集与监控（可接入 Google Analytics、自建或甲方指定平台）：
  - **LCP**（Largest Contentful Paint）：建议首屏 LCP &lt; 2.5s。
  - **FID**（First Input Delay）或 **INP**：建议 &lt; 100ms。
  - **CLS**（Cumulative Layout Shift）：建议 &lt; 0.1。
- **首屏加载**：可补充 **FCP**（First Contentful Paint）或 **FMP**（First Meaningful Paint）的优化目标（如 FCP &lt; 1.8s），与代码分割、关键资源预加载配合。
- **实施**：在应用根组件或路由入口注入上报逻辑，在开发/测试环境可输出到控制台便于调试。

### 8.3 安全与鉴权的边缘场景

#### Token 过期与刷新机制

- **约定**：需在设计中明确二选一或组合策略：
  - **方案 A — 强制重新登录**：Access Token 过期后，接口返回 401，前端清除本地 Token 并跳转 `/login?redirect=当前 URL`，用户登录后回跳。实现简单，但长时间停留会中断操作。
  - **方案 B — 无感续期**：后端颁发 Refresh Token（存 HttpOnly Cookie 或安全存储），Access Token 过期前或收到 401 时，前端调用刷新接口换取新 Access Token，成功后重试原请求；仅当 Refresh Token 也失效时再跳转登录。需后端支持 Refresh Token 及刷新接口。
- **建议**：若为 Web 长停留场景，优先采用方案 B 或「短期 Token + 静默刷新」；若为短会话，方案 A 即可。文档中需写明选用方案及刷新触发时机（如请求前检查过期时间、或统一在 401 拦截器中尝试刷新一次再决定是否跳转登录）。

#### 防 XSS 与 CSRF

- **XSS**：对所有**用户可见的、来自后端或用户输入的内容**进行转义后再渲染；富文本若允许 HTML，需使用受控的富文本库并按白名单标签/属性过滤。避免将未转义内容写入 `innerHTML` 或 `dangerouslySetInnerHTML`。
- **CSRF**：若鉴权改为或部分使用 **Cookie**（如 Session 或 HttpOnly Cookie 存 Token），后端须开启 CSRF 防护（如 SameSite 属性、CSRF Token、或双重 Cookie 校验）；若仅使用 Authorization Header + localStorage 且无 Cookie 鉴权，则 CSRF 风险较低，但需在文档中明确当前鉴权方式及对应防护要求。

### 8.4 部署与 DevOps 流程

#### CDN 与缓存策略

- **生产环境**：建议静态资源（JS、CSS、带 hash 的 chunk）走 **CDN**，缩短首屏加载时间。
- **缓存约定**：
  - **带 hash 的静态资源**（如 `assets/xxx.[hash].js`）：可设置长期缓存（如 `max-age=31536000, immutable`），依赖 Vite 等构建产出 hash 文件名。
  - **`index.html`**：不设长缓存，建议 `no-cache` 或短 `max-age` 配合协商缓存（ETag/Last-Modified），确保用户能及时拿到最新入口以加载新 chunk。
- **版本更新**：前端发版后，新会话会通过最新 index.html 拉取新资源；老会话可能仍缓存旧 HTML，待下次刷新或重新打开即更新。

#### 环境隔离与环境变量管理

- **环境划分**：建议至少区分 **开发（dev）**、**测试/预发（staging）**、**生产（prod）**；若有甲方独立测试环境可单独增加。
- **配置隔离**：
  - 前端：`VITE_API_BASE_URL`、`VITE_APP_TITLE` 等按环境在构建时注入（.env.development / .env.staging / .env.production），构建产物与环境一一对应，避免在生产构建中混入测试 API 地址。
  - 后端：API 的 CORS、数据库、日志级别等通过 Spring Profile 或环境变量区分，与前端环境对应（如 staging 前端对 staging 后端）。
- **规范**：在项目 README 或运维文档中列出各环境域名、构建命令、部署路径及环境变量清单，便于开发与运维对照执行。

## 九、可选增强（按需）

- **PWA**：manifest + Service Worker，支持「添加到主屏」、离线缓存，使移动端体验更接近 APP。
- **首屏与性能**：代码分割已按路由/ chunk 拆分时，确保首屏路由仅加载必要 chunk；可配合预加载或懒加载优化。
- **分析**：对访问来源、设备、主要路径做统计，便于后续优化 Web 端体验。

## 十、小结

从「APP 形态」到「移动端可访问的网页」，设计上的核心改动可归纳为：

1. **以 URL 为粒度的信息架构**：每个主要视图有唯一路径，支持刷新、收藏、分享与深度链接。  
2. **路由与导航一致**：底部 Tab（或大屏下的替代导航）与路由同步，返回与浏览器历史一致。  
3. **多端布局与安全区域**：在保留移动端体验的前提下，明确大屏与横屏策略，统一安全区域。  
4. **访问与部署**：固定域名、HTTPS、生产 CORS 与 API 配置，以及登录后回跳等 Web 化体验。  

在实现时，可先落地「路由 + URL 与 Tab 同步 + 部署与 CORS」，再按需迭代响应式、PWA 和 SEO，从而在不参考既有文档的前提下，系统性地完成 APP 到移动端 Web 的设计改造。

| 议题 | 已确定约定 |
|------|------------|
| **人员分工** | **甲**：路由与 Tab 同步、404、返回与深度链接（附录 A）；**乙**：布局、响应式、安全区域（附录 B）及体验与多页感知（附录 C）；**丙**：鉴权、登录回跳、Token 策略（附录 D）及后端 CORS、登录/刷新接口（G1、G2）。若 3 人均为前端，则 G1/G2 在联调阶段与后端统一对接。 |
| **公共基础** | 由**甲**先搭好路由表、路由常量（`src/constants/routes.ts`）及底部 Tab 与 path 映射，**乙、丙**在此基础上开发各自模块，避免多人同时改 `App.tsx` 或路由配置。 |
| **依赖顺序** | 先交付「路由壳 + 鉴权与 401/redirect」（A1～A4、A9 + D1、G2），再并行做 404（A5～A6）、各页 title（C2）、分享（C3）及布局（B）。 |

### 11.2 共享约定（必须统一）

| 议题 | 已确定约定 |
|------|------------|
| **路由路径** | 与文档一致。根路径 `/` 重定向至 `/chat`。路径：`/chat`、`/knowledge`、`/tickets`、`/tickets/:id`、`/profile`、`/login`、`/register`。`/knowledge/:id` 暂不启用，待产品确认后再加。代码中**禁止手写路径字符串**，统一从 `src/constants/routes.ts` 引用常量（如 `ROUTES.CHAT`、`ROUTES.TICKET_DETAIL`）。 |
| **redirect 参数名** | 使用 **`redirect`**。跳转登录时 URL 为 `?redirect=encodeURIComponent(当前完整 URL)`；登录成功后读取 `redirect` 并跳转，无则跳转 `/chat`。前后端统一使用该参数名。 |
| **API 契约** | 后端在开发前提供工单详情等接口的请求/响应格式及错误码说明（如 404 表示资源不存在）。前端按该契约解析 404 并展示 404 页或「工单不存在」等文案。 |
| **环境变量名** | 前端仅使用 **`VITE_API_BASE_URL`**（必选）。其余 `VITE_*` 若有再补充。后端 CORS 的 `allowed-origins` 开发阶段至少包含 `http://localhost:5173`。 |
| **Token 过期策略** | 采用**方案 A**：401 时清除本地 Token 并跳转 `/login?redirect=当前 URL`。暂不实现 Refresh Token 无感续期，后续如需再迭代。 |
| **历史栈边界** | 当 `navigate(-1)` 会离开本站时，**不**弹窗提示，改为跳转到当前资源的**上一级路径**（如从 `/tickets/123` 回到 `/tickets`）。该逻辑由甲统一实现。 |

### 11.3 分支与合并策略

| 议题 | 已确定约定 |
|------|------------|
| **分支命名** | 使用 **`feature/模块名`**：`feature/路由与404`、`feature/布局与响应式`、`feature/鉴权与回跳`（与分工对应）。禁止使用无含义分支名。 |
| **同步节奏** | **每 1～2 天**从主分支拉取一次并解决冲突，避免长期不合并导致大冲突。 |
| **谁合并主分支** | 合并前需**本地自测通过**。合入主分支前由**联调负责人**做一次集成验证（拉主分支 + 本分支，本地跑通主流程），通过后再合入；或由联调负责人统一合并到主分支，开发仅提交到各自 feature 分支。 |

### 11.4 代码与风格

| 议题 | 已确定约定 |
|------|------------|
| **路由与页面文件位置** | 路由配置集中在 **`App.tsx`** 或单独 **`src/routes/index.tsx`**（二选一，由甲定）。**新页面**统一放在 **`src/pages/`**（如 `ChatPage.tsx`、`TicketDetailPage.tsx`），与现有 `components/` 可并存，现有页面可逐步迁入 `pages/`。404 页、登录页、注册页均放在 `src/pages/`。 |
| **路由常量位置** | **`src/constants/routes.ts`**。导出 `ROUTES` 对象及 Tab 与 path 的映射表，所有人从此文件引用，禁止在组件中手写路径。 |
| **命名** | 页面组件与路由一一对应：`ChatPage`、`TicketDetailPage`、`NotFoundPage`、`LoginPage`、`RegisterPage`。路由常量与 path 对应：`ROUTES.CHAT`、`ROUTES.TICKET_DETAIL` 等。共享布局组件命名为 `*Layout`（如 `MainLayout`）。 |

### 11.5 联调与提测

| 议题 | 已确定约定 |
|------|------------|
| **联调环境** | 前端各自本地运行（`npm run dev`，端口 5173）。后端**统一由一人起服务**或使用同一后端地址，地址约定为 **`http://localhost:8080`**（或由联调负责人指定）；前端通过 `.env.development` 中 `VITE_API_BASE_URL` 指向该地址。联调时所有人前端连同一后端。 |
| **完成定义** | 涉及**路由、鉴权、跨页跳转、接口请求**的改动，需联调人员在主分支（或集成分支）上**跑通相关主流程**后，方可视为该修改点完成。仅样式或单页内部逻辑可仅自测。 |
| **提测方式** | 联调人员从**主分支**拉取代码，本地执行 `npm run dev` 与后端服务进行验证。暂不要求固定内网地址；若后续需要再约定。 |
| **问题归属** | **路由、404、返回、Tab 与 URL 同步** → 甲；**布局、安全区域、响应式** → 乙；**登录、401、redirect、Token** → 丙；**接口 4xx/5xx、CORS** → 后端或丙。联调负责人做初次判断并指派。 |

## 附录：开发修改点清单

以下列出开发阶段需要落实的**每一个修改点**，供实现时逐项对照。部署相关（如 CDN、Nginx、多环境发布流程等）暂不列入。

### A. 路由与信息架构

| 序号 | 修改点 |
|------|--------|
| A1 | 引入 React Router（或等价方案），在项目中配置路由与根组件。 |
| A2 | 定义路由表：`/` 或 `/chat`（问答）、`/knowledge`、`/knowledge/:id`（若产品需要）、`/tickets`、`/tickets/:id`、`/profile`、`/login`、`/register`。 |
| A3 | 实现路由与底部 Tab 双向同步：用户点击 Tab 时更新 URL；通过链接或浏览器前进/后退进入时，根据 URL 展示对应页面并高亮对应 Tab。 |
| A4 | 确保刷新页面后根据当前 URL 恢复对应页面并重新拉取该页所需数据。 |
| A5 | 在路由表中增加通配路由（如 `path: '*'`），渲染自定义 404 组件，并提供「返回首页」「返回工单列表」等入口。 |
| A6 | 对 `/tickets/:id` 等依赖后端数据的路由：请求详情接口若返回 404，展示 404 页或「工单不存在」等文案，不静默跳转首页。 |
| A7 | 所有页面内「返回」按钮统一使用 `navigate(-1)`（或等价），与浏览器后退行为一致。 |
| A8 | 当历史栈上一页为站外时，对 `navigate(-1)` 做边界处理：二选一并实现——提示「将离开当前站点」或改为跳转上一级路径（如 `/tickets/123` → `/tickets`）。 |
| A9 | 未登录用户访问需登录的 URL 时，跳转 `/login?redirect=当前完整 URL`；登录成功后根据 `redirect` 参数跳回原页面，无则回默认页。 |
| A10 | 支持深度链接：直接打开 `/tickets/:id` 等 URL 时正确展示详情，并完成鉴权与 404 处理。 |

### B. 布局与响应式

| 序号 | 修改点 |
|------|--------|
| B1 | 确定并实现断点策略（如 768px、1024px），在导航、内容宽度、触控区域上统一使用。 |
| B2 | 大屏下内容区设置更大 max-width（如 672px 或 896px），避免单行过长。 |
| B3 | 大屏导航方案二选一并实现：方案 A 保留底部 Tab；方案 B 改为顶栏或侧边栏导航。 |
| B4 | 横屏与键盘弹起时，对输入框、底部按钮等使用 `env(safe-area-inset-*)` 或 viewport 单位，保证可视与可点。 |
| B5 | 全站统一使用安全区域（刘海、底部横条、横屏左右），避免关键控件被遮挡。 |
| B6 | 底部 Tab、主要按钮等可点击区域不小于 44×44 逻辑像素。 |
| B7 | 若支持桌面端：为可交互元素增加鼠标悬停与键盘焦点样式，与触控样式不冲突。 |

### C. 体验与多页感知

| 序号 | 修改点 |
|------|--------|
| C1 | 登录/注册以全屏页或弹层呈现时，明确与历史栈的关系（压栈或替换），避免用户需多次点后退才能关闭。 |
| C2 | 为每个路由设置对应的 `document.title`（及可选的 meta description），便于多标签与收藏显示（如「工单 #123 - 售后支持」）。 |
| C3 | 分享功能使用当前页面 URL，确保他人打开可进入同一页（权限由后端与鉴权逻辑保证）。 |

### D. 鉴权与安全

| 序号 | 修改点 |
|------|--------|
| D1 | 未授权场景统一为跳转 `/login?redirect=当前 URL`（若保留弹窗方案，需在文档中约定使用场景）。 |
| D2 | Token 过期策略：选定方案 A（401 即跳转登录 + redirect）或方案 B（Refresh Token 无感续期），并实现对应逻辑（请求前检查过期或 401 拦截器中尝试刷新一次再决定是否跳转）。 |
| D3 | 所有用户可见且来自后端或用户输入的内容，渲染前做转义；富文本若允许 HTML，使用受控库并按白名单标签/属性过滤；禁止未转义内容写入 `innerHTML` 或 `dangerouslySetInnerHTML`。 |
| D4 | 若采用 Cookie 鉴权（或部分接口用 Cookie）：后端实现 CSRF 防护（SameSite / CSRF Token / 双重 Cookie 等）；在文档中写明当前鉴权方式及对应防护要求。 |

### E. 浏览器兼容与性能

| 序号 | 修改点 |
|------|--------|
| E1 | 在项目中配置 `browserslist` 或 Vite `build.target`，与兼容性矩阵（Chrome 80+、Safari 14+、Firefox 80+、Edge 80+）一致。 |
| E2 | 维护「兼容性检查清单」，在目标机型（尤其 iOS 不同机型）上回归关键页面与安全区域表现。 |
| E3 | 在应用根组件或路由入口注入 Web Vitals（LCP、FID/INP、CLS）及首屏 FCP/FMP 的上报逻辑；开发/测试环境可将指标输出到控制台便于调试。 |

### F. 前端构建与配置（开发侧）

| 序号 | 修改点 |
|------|--------|
| F1 | 若前端需支持非根路径访问，在 Vite 与路由中配置 `base`，避免资源 404。 |
| F2 | 使用 `VITE_API_BASE_URL` 等环境变量配置 API 地址，去除对 localhost 的硬编码；`index.html` 中的 preconnect/dns-prefetch 随环境配置或移除硬编码域名。 |

### G. 后端（开发侧）

| 序号 | 修改点 |
|------|--------|
| G1 | 后端 CORS `allowed-origins` 包含前端实际访问源（开发阶段至少包含 `http://localhost:5173` 等本地地址）。 |
| G2 | 若采用「登录后回跳」：登录接口或前端从 URL 的 `redirect` 参数读取并在登录成功后跳转；若采用 Token 刷新方案，后端提供刷新接口并与前端约定请求/响应格式及刷新触发条件。 |
