# 后台流式继续：多对话并发设计文档

> 版本：1.0 · 日期：2026-03-06  
> 基于：`conversation-switching-design.md` v1.0 的演进

---

## 一、需求变更说明

### 旧需求（v1.0）
> 用户新建对话或切换历史时，**立即取消**当前 SSE 流，旧对话的截断回复丢失。

### 新需求（v2.0）
> 用户新建对话后，**旧对话继续在后台完成**；  
> 用户切回旧对话时，能看到**实时进度**（若仍在生成）或**完整回复**（若已完成）。

---

## 二、核心架构变化

### v1.0：单对话槽位

```
ChatPage
 ├── messages         (单一消息列表)
 ├── aiThinking       (单一状态)
 ├── cancelChatRef    (单一取消函数)
 └── currentSessionId (单一会话ID)

切换时：cancel旧流 → 清空messages → 加载新对话
```

### v2.0：多对话槽位（Background Slot 模式）

```
ChatPage
 ├── [当前视图槽位]
 │    ├── messages         (当前对话的消息列表，React state，驱动渲染)
 │    ├── aiThinking       (当前对话是否在生成)
 │    ├── cancelChatRef    (当前视图的 cancel 函数)
 │    └── currentSessionId (当前视图的会话ID)
 │
 ├── backgroundSlotsRef: Map<slotKey, BackgroundSlot>
 │    ├── slot-A (旧对话A，仍在后台生成)
 │    │    ├── messages[]     (累积中的消息，直接写入 ref，不触发渲染)
 │    │    ├── isStreaming     (true/false)
 │    │    ├── cancelFn       (旧对话的 cancel 函数)
 │    │    ├── activeConvId   (旧对话的 conversationId)
 │    │    └── sessionId      (旧对话的后端 conversationId)
 │    └── slot-B (另一个已完成的旧对话)
 │
 └── currentSlotKeyRef: string  ("当前视图"对应哪个 slotKey)

切换时：保存当前为 backgroundSlot → 切换视图到新slot → 旧流继续运行
```

## 三、数据结构定义

```typescript
/** 后台对话槽位（存储于 ref，不触发 React 渲染） */
interface BackgroundSlot {
  messages: Message[];           // 后台累积的消息列表
  sessionId: string | null;      // 后端分配的 conversationId
  isStreaming: boolean;          // 是否仍在后台生成
  isPaused: boolean;             // 是否被用户暂停
  cancelFn: (() => void) | null; // 中断 SSE 的函数
  activeConvId: string | null;   // 后端 conversationId（用于 abort 接口）
  stepInterval: ReturnType<typeof setInterval> | null;
  isHistoryConversation: boolean;
  followUpQuestions: string[];
  currentMessageId: string | null;
}

// slotKey：前端内部唯一标识符
// - 新建对话时生成：`slot-${Date.now()}`
// - 已知 sessionId 时，可用 sessionId 作为 key
type SlotKey = string;
```

```typescript
// 新增的 Ref（在 ChatPage 顶层声明）
const backgroundSlotsRef = useRef<Map<SlotKey, BackgroundSlot>>(new Map());
const currentSlotKeyRef  = useRef<SlotKey>('initial');
```

## 四、流式回调路由机制

**核心设计**：每个 SSE 流启动时，捕获 `slotKey`。回调触发时，对比 `currentSlotKeyRef.current` 决定路由到哪里。

```typescript
// handleSendMessage 内部，流启动前：
const slotKey = currentSlotKeyRef.current;

// 路由函数：当前视图 → setMessages (触发渲染)；后台 → 直接写 ref
const routeMessages = (updater: (prev: Message[]) => Message[]) => {
  if (currentSlotKeyRef.current === slotKey) {
    setMessages(updater);       // 当前视图，驱动 React 渲染
  } else {
    const slot = backgroundSlotsRef.current.get(slotKey);
    if (slot) slot.messages = updater(slot.messages); // 后台，静默写 ref
  }
};

const routeThinking = (val: boolean) => {
  if (currentSlotKeyRef.current === slotKey) {
    setAiThinking(val);
  } else {
    const slot = backgroundSlotsRef.current.get(slotKey);
    if (slot) slot.isStreaming = val;
  }
};

// 在 SSE 回调中，将原 setMessages(...) 替换为 routeMessages(...)
// 将原 setAiThinking(...) 替换为 routeThinking(...)
```

## 五、核心流程（v2.0）

### 5.1 正常对话（不变）

```
用户发送消息
  │
  ├─ 捕获 slotKey = currentSlotKeyRef.current
  ├─ 启动 SSE，cancelFn 写入当前 cancelChatRef
  ├─ SSE 回调使用 routeMessages / routeThinking（当前视图）
  └─ message_end → 后端 saveAiResponse() 入库
```

### 5.2 新建对话（核心变化）

```
用户点击"新建对话"（旧对话 A 仍在生成）
  │
  ├─ 将当前状态保存到 backgroundSlotsRef
  │   ├── backgroundSlotsRef.set('slot-A', {
  │   │     messages: [...当前messages],
  │   │     sessionId: currentSessionId,
  │   │     isStreaming: true,         ← 标记仍在生成
  │   │     cancelFn: cancelChatRef.current,
  │   │     activeConvId: activeConvIdRef.current,
  │   │     ...
  │   │   })
  │   └── 此时 slot-A 的流式回调仍在运行
  │        └── 回调内 currentSlotKeyRef.current ≠ 'slot-A'
  │             → routeMessages 会静默写入 backgroundSlotsRef
  │
  ├─ 生成新 slotKey：'slot-B'
  ├─ currentSlotKeyRef.current = 'slot-B'
  ├─ setMessages([欢迎消息])    ← 新对话的视图
  ├─ setAiThinking(false)       ← 新对话按钮恢复"发送"
  ├─ setCurrentSessionId(null)
  │
  └─ ⚠️  NOT cancel slot-A 的流 → 旧流继续在后台完成
```

### 5.3 切回旧对话（从历史列表）

```
用户点击历史列表中的对话 A（slot-A 在 backgroundSlotsRef 中）
  │
  ├─ 检查 backgroundSlotsRef.has(conversationId) ?
  │
  ├─ [分支 A：对话 A 仍在后台]
  │   ├─ 将当前视图（slot-B）保存到 backgroundSlotsRef
  │   ├─ 从 backgroundSlotsRef 取出 slot-A
  │   ├─ currentSlotKeyRef.current = 'slot-A'   ← 路由指向 slot-A
  │   ├─ setMessages(slot-A.messages)            ← 恢复已有消息（含流式中间状态）
  │   ├─ setAiThinking(slot-A.isStreaming)       ← 若仍在生成，显示暂停按钮
  │   ├─ setCurrentSessionId(slot-A.sessionId)
  │   ├─ cancelChatRef.current = slot-A.cancelFn
  │   ├─ activeConvIdRef.current = slot-A.activeConvId
  │   └─ 后续 SSE 回调路由到 currentSlotKeyRef ← 自动路由到 setMessages
  │
  └─ [分支 B：对话 A 不在后台（已完成或是纯历史）]
      ├─ GET /ai/conversations/{id} 拉取数据库消息
      └─ 正常加载（与 v1.0 一致）
```

### 5.4 后台对话完成（无用户干预）

```
slot-A 后台流持续接收 SSE
  │
  ├─ 每个 message 片段：routeMessages → 静默写 slot-A.messages
  │
  └─ message_end 到来
      ├─ 后端 saveAiResponse() 入库 ✅（与 v1.0 完全一致）
      ├─ routeThinking(false) → 静默写 slot-A.isStreaming = false
      ├─ slot-A.cancelFn = null
      └─ 若用户此时已切回 slot-A：
           cancelChatRef.current 已置为 null
           setAiThinking(false) 正常触发（因为此时 currentSlotKeyRef = 'slot-A'）
```

### 5.5 暂停后台对话（用户操作）

```
⚠️  当前设计：暂停按钮只对"当前视图"的对话有效
    用户切回旧对话后，若仍在生成 → 可点暂停 → 触发 handleStopGeneration()
    后台对话不提供遥控暂停入口（可在历史列表加指示器，点击后先切回再暂停）
```

### 5.6 切换 Tab（keep-alive，不变）

```
ChatPage 组件保持挂载（display:none）
所有 SSE 连接（含后台 slot）均继续运行
```

## 六、后端能力核查（针对 v2.0 新需求）

### 6.1 核心问题：后端是否支持同一用户并发多个 SSE 流？

已查阅 `AiService.java` + `DifyClient.java`，结论如下：

**`AiService.java`**：
```java
// 每次 chat() 调用创建独立的 SseEmitter，互不共享
SseEmitter emitter = new SseEmitter(difyConfig.getTimeout());
// taskId = conversationId（各对话独立）
final String taskId = conversation.getConversationId();
```

**`DifyClient.java`**：
```java
// ConcurrentHashMap：线程安全，支持多 taskId 并发
private final ConcurrentHashMap<String, HttpURLConnection> activeConnections
    = new ConcurrentHashMap<>();

// newCachedThreadPool：按需创建线程，无数量上限，支持任意并发
private final ExecutorService executorService = Executors.newCachedThreadPool();

// cancelStream：只移除指定 taskId，不影响其他流
public void cancelStream(String taskId) {
    HttpURLConnection conn = activeConnections.remove(taskId); // 精准定位
    if (conn != null) conn.disconnect();
}
```

**结论：✅ 后端完全支持并发多流，已通过代码确认**
- `ConcurrentHashMap` 保证多线程安全写入
- 每个 `taskId` 对应一个独立的 `HttpURLConnection`，互不干扰
- `cancelStream` 精准移除，不会误杀其他对话的流

### 6.2 全面能力核查表（已全部确认）

| 能力 | v1.0 状态 | v2.0 需求 | 是否满足 | 依据 |
|---|---|---|---|---|
| 并发多个 SSE 流（同一用户） | 未使用 | **新增核心需求** | ✅ **已确认** | `ConcurrentHashMap` + `newCachedThreadPool` |
| 各流独立取消 | ✅ | 必须 | ✅ **已确认** | `cancelStream` 按 taskId 精准移除 |

| 主动停止指定对话 | ✅ | 必须 | ✅ | `POST /conversations/{id}/abort` |

### 6.3 已排除的风险点

| 风险点 | 确认结果 |
|---|---|

### 7.2 修改 `resetToNewChat()`

```typescript
const resetToNewChat = () => {
  // 1. 将当前状态保存为后台槽位（不取消流）
  const oldSlotKey = currentSlotKeyRef.current;
  if (aiThinking && cancelChatRef.current) {
    backgroundSlotsRef.current.set(oldSlotKey, {
      messages: [...messages],          // 快照当前消息
      sessionId: currentSessionId,
      isStreaming: true,
      isPaused: false,
      cancelFn: cancelChatRef.current,  // 保留 cancel 能力
      activeConvId: activeConvIdRef.current,
      stepInterval: stepIntervalRef.current,
      isHistoryConversation: false,
      followUpQuestions: [...followUpQuestions],
      currentMessageId: currentMessageId,
    });
    // 清空当前引用（不 cancel）
    cancelChatRef.current = null;
    activeConvIdRef.current = null;
    stepIntervalRef.current = null;
  }

  // 2. 切换到新槽位
  const newSlotKey = `slot-${Date.now()}`;
  currentSlotKeyRef.current = newSlotKey;

  // 3. 重置视图状态（原有逻辑）
  setAiThinking(false);
  setThinkingPaused(false);
  setMessages([{ id: Date.now().toString(), type: 'ai', content: '...', timestamp: new Date() }]);
  setCurrentSessionId(null);
  setFollowUpQuestions([]);
  setCurrentMessageId(null);
  setTicketCreated(false);
  setShowTicketPrompt(false);
  setIsHistoryConversation(false);
  setHasCreatedNewChat(true);
};
```

### 7.3 修改 `loadConversationDetail()`

```typescript
const loadConversationDetail = async (conversationId: string) => {
  // 1. 检查是否有此对话的后台槽位
  const bgSlot = backgroundSlotsRef.current.get(conversationId);
  if (bgSlot) {
    // 将当前视图保存为后台槽（若当前也在生成）
    saveCurrentToBackground();

    // 切换到后台槽
    currentSlotKeyRef.current = conversationId;
    backgroundSlotsRef.current.delete(conversationId);

    // 恢复后台槽的状态到视图
    setMessages(bgSlot.messages);
    setCurrentSessionId(bgSlot.sessionId);
    setAiThinking(bgSlot.isStreaming);
    setThinkingPaused(bgSlot.isPaused);
    setFollowUpQuestions(bgSlot.followUpQuestions);
    setCurrentMessageId(bgSlot.currentMessageId);
    setIsHistoryConversation(bgSlot.isHistoryConversation);
    cancelChatRef.current = bgSlot.cancelFn;
    activeConvIdRef.current = bgSlot.activeConvId;
    stepIntervalRef.current = bgSlot.stepInterval;
    return;  // 不走网络请求
  }

  // 2. 无后台槽位：原有流程（取消当前流 + 拉接口）
  saveCurrentToBackground(/* 是否取消 */ true);
  // ... 原有的 GET /ai/conversations/{id} 逻辑
};
```

## 九、后端能力总结

### ✅ 后端完全满足 v2.0 的新需求，无需修改任何后端代码。

| 需求点 | 后端状态 |

| 确认点 | 风险 | 处置 |
