# 实体类设计文档

## 文档说明

本文档详细说明清洁机器人智能指南平台后端系统的所有实体类（Entity）设计，包括字段定义、数据类型、约束条件、实体关系等。

---

## 一、实体类概览

系统共包含 **7个核心实体类**：

| 实体类 | 表名 | 说明 |
|--------|------|------|
| User | users | 用户实体 |
| Conversation | conversations | AI对话会话实体 |
| Message | messages | 对话消息实体 |
| Ticket | tickets | 工单实体 |
| MediaFile | media_files | 媒体文件/知识文档实体 |
| TicketFeedback | ticket_feedbacks | 工单反馈实体（赞/踩） |
| MessageFeedback | message_feedbacks | 消息反馈实体（赞/踩） |

---

## 二、实体关系图

```
User (用户)
  │
  ├── 1:N → Conversation (会话)
  │       └── 1:N → Message (消息)
  │                   └── N:M → MessageFeedback (消息反馈)
  │                             ↑
  │                             └── User (用户)
  │
  └── 1:N → Ticket (工单)
          └── 1:1 → TicketFeedback (工单反馈)

MediaFile (媒体文件) - 独立实体，无关联
```

## 三、实体类详细设计

### 3.1 User（用户实体）

**表名：** `users`

**说明：** 存储系统用户信息，支持多种角色（经销商、厂商、普通用户）

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键（自增） |
| userId | String(50) | UNIQUE, NOT NULL | 业务ID，如 "U12345" |
| phone | String(20) | UNIQUE, NOT NULL | 手机号（登录账号） |
| password | String(100) | NULL | 密码（加密后存储） |
| nickname | String(50) | NULL | 用户昵称 |
| avatar | String(500) | NULL | 头像URL |
| memberTag | String(50) | NULL | 会员标签，如"高级用户" |
| role | String(20) | NULL | 角色：dealer(经销商), manufacturer(厂商), user(普通用户) |
| createdAt | LocalDateTime | NOT NULL | 创建时间（自动设置） |
| updatedAt | LocalDateTime | NOT NULL | 更新时间（自动更新） |

#### 特殊设计

- **双重ID设计**：`id`（数据库主键）+ `userId`（业务ID）
- **自动时间戳**：使用 `@PrePersist` 和 `@PreUpdate` 自动维护时间
- **唯一性约束**：`userId` 和 `phone` 均唯一

#### 代码示例

```java
@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(unique = true, nullable = false, length = 50)
    private String userId;
    
    // ... 其他字段
}
```

### 3.2 Conversation（会话实体）

**表名：** `conversations`

**说明：** 存储AI对话会话信息，一个用户可以有多个会话

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键 |
| conversationId | String(100) | UNIQUE, NOT NULL | 业务ID，如 "conv_12345678" |
| user | User | FK, NOT NULL | 关联用户（多对一关系） |
| title | String(200) | NULL | 会话标题（从第一条消息提取） |
| createdAt | LocalDateTime | NOT NULL | 创建时间 |
| updatedAt | LocalDateTime | NOT NULL | 更新时间 |

#### 关系定义

- **多对一（Many-to-One）**：多个会话属于一个用户
  ```java
  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "user_id", nullable = false)
  private User user;
  ```

#### 特殊设计

- **懒加载**：使用 `FetchType.LAZY` 避免不必要的用户信息加载
- **业务ID**：`conversationId` 用于前端展示和API调用

### 3.3 Message（消息实体）

**表名：** `messages`

**说明：** 存储对话中的每条消息，包括用户消息和AI回复

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键 |
| conversation | Conversation | FK, NOT NULL | 关联会话（多对一关系） |
| role | MessageRole | NOT NULL | 消息角色：user 或 assistant |
| content | String(TEXT) | NULL | 消息内容（支持长文本） |
| timestamp | LocalDateTime | NOT NULL | 消息时间戳 |

#### 枚举类型

```java
public enum MessageRole {
    user,        // 用户消息
    assistant    // AI助手回复
}
```

#### 关系定义

- **多对一（Many-to-One）**：多条消息属于一个会话
  ```java
  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "conversation_id", nullable = false)
  private Conversation conversation;
  ```

#### 特殊设计

- **枚举存储**：使用 `@Enumerated(EnumType.STRING)` 将枚举存储为字符串
- **大文本支持**：`content` 使用 `TEXT` 类型支持长文本
- **自动时间戳**：`@PrePersist` 自动设置时间戳

### 3.4 Ticket（工单实体）

**表名：** `tickets`

**说明：** 存储服务工单信息，支持状态流转和优先级管理

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键 |
| ticketId | String(50) | UNIQUE, NOT NULL | 业务ID，如 "WO20240120001" |
| user | User | FK, NOT NULL | 关联用户（多对一关系） |
| title | String(200) | NOT NULL | 工单标题 |
| description | String(TEXT) | NULL | 工单描述 |
| status | TicketStatus | NOT NULL | 工单状态（枚举） |
| priority | TicketPriority | NOT NULL | 优先级（枚举） |
| relatedChatId | String(100) | NULL | 关联的对话ID |
| attachmentUrls | List<String> | NULL | 附件URL列表（集合字段） |
| engineerName | String(100) | NULL | 工程师姓名 |
| estimatedTime | String(100) | NULL | 预计处理时间 |
| comments | String(TEXT) | NULL | 备注/处理意见 |
| createdAt | LocalDateTime | NOT NULL | 创建时间 |
| updatedAt | LocalDateTime | NOT NULL | 更新时间 |

#### 枚举类型

**工单状态（TicketStatus）：**
```java
public enum TicketStatus {
    pending,      // 待处理
    processing,   // 处理中
    completed,    // 已完成
    cancelled     // 已取消
}
```

**优先级（TicketPriority）：**
```java
public enum TicketPriority {
    low,      // 低
    medium,   // 中
    high      // 高
}
```

#### 关系定义

- **多对一（Many-to-One）**：多个工单属于一个用户
  ```java
  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "user_id", nullable = false)
  private User user;
  ```

- **集合字段（ElementCollection）**：附件URL列表存储到独立表
  ```java
  @ElementCollection
  @CollectionTable(name = "ticket_attachments", 
                   joinColumns = @JoinColumn(name = "ticket_id"))
  @Column(name = "attachment_url")
  @Builder.Default
  private List<String> attachmentUrls = new ArrayList<>();
  ```
  
  **生成的表结构：**
  - 主表：`tickets`
  - 关联表：`ticket_attachments`
    - `ticket_id` (FK)
    - `attachment_url` (VARCHAR)

#### 特殊设计

- **默认状态**：创建时自动设置为 `pending`
- **业务ID格式**：`WO` + 日期(8位) + 序号(3位)，如 `WO20240120001`

### 3.6 TicketFeedback（工单反馈实体）

**表名：** `ticket_feedbacks`

**说明：** 存储用户对工单处理的反馈（赞/踩）

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键 |
| ticket | Ticket | FK, NOT NULL, UNIQUE | 关联工单（一对一关系） |
| type | FeedbackType | NOT NULL | 反馈类型（枚举） |
| comment | String(TEXT) | NULL | 反馈意见 |
| createdAt | LocalDateTime | NOT NULL | 创建时间 |

#### 枚举类型

```java
public enum FeedbackType {
    like,      // 赞
    dislike    // 踩
}
```

#### 关系定义

- **一对一（One-to-One）**：一个工单只能有一个反馈
  ```java
  @OneToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "ticket_id", nullable = false, unique = true)
  private Ticket ticket;
  ```

#### 特殊设计

- **唯一性约束**：确保每个工单只能有一个反馈记录
- **可选反馈**：`comment` 字段可选，用户可以不填写意见

### 3.7 MessageFeedback（消息反馈实体）

**表名：** `message_feedbacks`

**说明：** 存储用户对AI消息的评价（赞/踩），支持多用户对同一条消息的评价

#### 字段定义

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 数据库主键 |
| message | Message | FK, NOT NULL | 关联消息（多对一关系） |
| user | User | FK, NOT NULL | 关联用户（多对一关系） |
| type | FeedbackType | NOT NULL | 反馈类型（枚举） |
| comment | String(TEXT) | NULL | 反馈意见（可选） |
| createdAt | LocalDateTime | NOT NULL | 创建时间 |
| updatedAt | LocalDateTime | NOT NULL | 更新时间 |

#### 枚举类型

```java
public enum FeedbackType {
    like,      // 赞
    dislike    // 踩
}
```

#### 关系定义

- **多对一（Many-to-One）**：多个反馈属于一条消息
  ```java
  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "message_id", nullable = false)
  private Message message;
  ```

- **多对一（Many-to-One）**：多个反馈属于一个用户
  ```java
  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "user_id", nullable = false)
  private User user;
  ```

- **唯一性约束**：同一用户对同一条消息只能评价一次
  ```java
  @Table(name = "message_feedbacks", 
         uniqueConstraints = @UniqueConstraint(columnNames = {"message_id", "user_id"}))
  ```

#### 特殊设计

- **多对多关系实现**：通过中间表实现用户和消息的多对多评价关系
- **唯一性约束**：`(message_id, user_id)` 组合唯一，确保用户对同一条消息只能评价一次
- **可更新**：用户可以先赞后踩，或先踩后赞（更新操作）
- **时间戳管理**：记录创建时间和更新时间，支持评价修改历史追踪

#### 业务场景

1. **用户评价消息**：用户可以对AI回复的消息进行赞/踩评价
2. **评价统计**：可以统计某条消息的赞数和踩数
3. **评价修改**：用户可以修改之前的评价（从赞改为踩，或反之）
4. **评价查询**：可以查询用户对某条消息的评价状态

#### 代码示例

```java
@Entity
@Table(name = "message_feedbacks", 
       uniqueConstraints = @UniqueConstraint(columnNames = {"message_id", "user_id"}))
public class MessageFeedback {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "message_id", nullable = false)
    private Message message;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User user;
    
    @Column(nullable = false, length = 20)
    @Enumerated(EnumType.STRING)
    private FeedbackType type;
    
    // ... 其他字段
}
```

## 五、设计原则与最佳实践

### 5.1 双重ID设计

所有实体都采用**双重ID**设计：
- **数据库主键（id）**：Long类型，自增，用于数据库内部关联
- **业务ID**：String类型，唯一，用于API接口和前端展示

**优势：**
- 业务ID不暴露数据库内部结构
- 支持更灵活的ID生成策略
- 便于分布式系统扩展

### 5.2 时间戳自动管理

所有实体都包含 `createdAt` 和 `updatedAt` 字段，通过JPA生命周期回调自动管理：

```java
@PrePersist
protected void onCreate() {
    createdAt = LocalDateTime.now();
    updatedAt = LocalDateTime.now();
}

@PreUpdate
protected void onUpdate() {
    updatedAt = LocalDateTime.now();
}
```

### 5.3 懒加载策略

所有关联关系都使用 `FetchType.LAZY`（懒加载）：

**优势：**
- 避免N+1查询问题
- 提高查询性能
- 按需加载关联数据

**注意：** 在Service层需要关联数据时，使用 `@EntityGraph` 或 `JOIN FETCH` 显式加载。

### 5.4 枚举类型存储

使用 `@Enumerated(EnumType.STRING)` 将枚举存储为字符串：

**优势：**
- 数据库可读性好
- 便于数据迁移和维护
- 支持枚举值变更

### 5.5 集合字段设计

使用 `@ElementCollection` 存储简单类型的集合（如 `List<String>`）：

**适用场景：**
- 附件URL列表
- 关联产品列表
- 标签列表

**注意：** 复杂对象集合应使用 `@OneToMany` 或 `@ManyToMany`。

## 七、业务ID生成规则

### 7.1 User.userId

**格式：** `U` + 时间戳后6位 + 随机4位

**示例：** `U123456ABCD`

**生成方法：**
```java
public static String generateUserId() {
    String timestamp = String.valueOf(System.currentTimeMillis());
    String suffix = timestamp.substring(timestamp.length() - 6);
    String random = UUID.randomUUID().toString().substring(0, 4).replace("-", "");
    return "U" + suffix + random.toUpperCase();
}
```

### 7.2 Conversation.conversationId

**格式：** `conv_` + UUID前8位

**示例：** `conv_12345678`

**生成方法：**
```java
public static String generateConversationId() {
    return "conv_" + UUID.randomUUID().toString().replace("-", "").substring(0, 8);
}
```

### 7.3 Ticket.ticketId

**格式：** `WO` + 日期(8位) + 序号(3位)

**示例：** `WO20240120001`

**生成方法：**
```java
public static String generateTicketId() {
    String date = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMdd"));
    String sequence = String.format("%03d", (int)(Math.random() * 1000));
    return "WO" + date + sequence;
}
```

### 7.4 MediaFile.fileId

**格式：** `KB` + UUID前6位（大写）

**示例：** `KB123456`

**生成方法：**
```java
public static String generateFileId() {
    return "KB" + UUID.randomUUID().toString().replace("-", "").substring(0, 6).toUpperCase();
}
```

## 八、实体类生命周期

### 8.1 JPA生命周期回调

所有实体类都实现了JPA生命周期回调方法：

#### @PrePersist（保存前）
- 设置 `createdAt` 和 `updatedAt` 为当前时间
- 设置默认值（如Ticket的status默认为pending）

#### @PreUpdate（更新前）
- 更新 `updatedAt` 为当前时间

### 8.2 实体状态

| 状态 | 说明 | 触发时机 |
|------|------|----------|
| Transient（瞬时） | 新创建的对象，未与数据库关联 | `new Entity()` |
| Managed（托管） | 已与数据库关联，在持久化上下文中 | `entityManager.persist()` 后 |
| Detached（分离） | 已从持久化上下文分离 | `entityManager.detach()` 或事务结束 |
| Removed（删除） | 标记为删除，将在事务提交时删除 | `entityManager.remove()` 后 |

## 九、查询优化建议

### 9.1 避免N+1查询

**问题场景：**
```java
// 错误示例：会导致N+1查询
List<Conversation> conversations = conversationRepository.findAll();
for (Conversation conv : conversations) {
    User user = conv.getUser(); // 每次循环都会查询一次数据库
}
```

**解决方案：**
```java
// 方案1：使用JOIN FETCH
@Query("SELECT c FROM Conversation c JOIN FETCH c.user WHERE c.user.id = :userId")
List<Conversation> findByUserIdWithUser(@Param("userId") Long userId);

// 方案2：使用@EntityGraph
@EntityGraph(attributePaths = {"user"})
List<Conversation> findByUser(User user);
```

### 9.2 批量操作优化

**批量插入：**
```java
@Modifying
@Query(value = "INSERT INTO messages (conversation_id, role, content, timestamp) VALUES ...", 
       nativeQuery = true)
void batchInsertMessages(List<Message> messages);
```

**批量更新：**
```java
@Modifying
@Query("UPDATE Ticket t SET t.status = :status WHERE t.id IN :ids")
void batchUpdateStatus(@Param("status") TicketStatus status, @Param("ids") List<Long> ids);
```

### 9.3 分页查询

```java
Pageable pageable = PageRequest.of(page, size, Sort.by("createdAt").descending());
Page<Conversation> conversations = conversationRepository.findByUser(user, pageable);
```

## 十、数据迁移与版本控制

### 10.1 JPA DDL策略

**配置（application.yml）：**
```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: update  # 开发环境：update，生产环境：validate
```

**策略说明：**
- `none`：不执行任何操作
- `validate`：验证表结构，不修改
- `update`：更新表结构（添加字段等）
- `create`：每次启动删除并重建表（仅开发）
- `create-drop`：启动时创建，关闭时删除（仅开发）

### 10.2 数据库迁移建议

**生产环境：**
- 使用 Flyway 或 Liquibase 进行数据库版本控制
- 禁用 `ddl-auto: update`
- 手动编写迁移脚本

**示例迁移脚本（Flyway）：**
```sql
-- V1__create_users_table.sql
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(50) UNIQUE NOT NULL,
    phone VARCHAR(20) UNIQUE NOT NULL,
    -- ... 其他字段
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);
```

## 十二、数据完整性约束

### 12.1 外键约束

所有外键关系都设置了 `nullable = false`，确保数据完整性：

- `conversations.user_id` → `users.id`（必填）
- `messages.conversation_id` → `conversations.id`（必填）
- `tickets.user_id` → `users.id`（必填）
- `ticket_feedbacks.ticket_id` → `tickets.id`（必填且唯一）
- `message_feedbacks.message_id` → `messages.id`（必填）
- `message_feedbacks.user_id` → `users.id`（必填）

## 十四、常见问题与解决方案

### 14.1 懒加载异常（LazyInitializationException）

**问题：** 在事务外访问懒加载的关联对象

**解决方案：**
1. 使用 `@Transactional` 保持事务
2. 使用 `JOIN FETCH` 预加载
3. 使用DTO投影避免加载实体

### 14.2 集合字段更新问题

**问题：** `@ElementCollection` 的集合更新可能不生效

**解决方案：**
```java
// 清空后重新添加
ticket.getAttachmentUrls().clear();
ticket.getAttachmentUrls().addAll(newUrls);
ticketRepository.save(ticket);
```

### 14.3 时间字段时区问题

**问题：** `LocalDateTime` 在不同时区可能显示不一致

**解决方案：**
- 统一使用UTC时间存储
- 前端根据用户时区转换显示
- 或使用 `ZonedDateTime` 存储时区信息
