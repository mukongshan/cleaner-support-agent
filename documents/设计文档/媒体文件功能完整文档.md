## 一、功能概述

### 1.1 业务场景

`MediaFile` 实体用于存储系统中的媒体文件和知识文档，包括：

- **销售资料**：PDF、PPT、Excel 等销售相关文档
- **产品资料**：产品图片、视频、规格说明等
- **维护资料**：维护视频、操作指南、故障排除文档
- **培训资料**：培训视频、教程文档等

### 1.2 支持的访问方式

- **Seafile**：通过 NJU Box (Seafile) 访问
- **本地文件**：存储在服务器本地
- **对象存储**：存储在 OSS 等云存储服务

### 1.3 核心功能

- 文件列表查询（按分类、关键词搜索）
- 文件详情查看
- 文件访问信息获取（预览链接和下载链接）
- 文件预览（支持在线预览）
- 文件下载
- 图片上传与展示（与图片识别功能集成）

- `repoId`：Seafile 资料库ID（从配置中读取，不需要每个文件都存储）

### 2.3 枚举类型

#### FileType（文件类型枚举）

```java
public enum FileType {
```

| `.xlsx`, `.xls` | `Excel` | ❌ 否 |
| `.pptx`, `.ppt` | `PPT` | ❌ 否 |

#### AccessMethod（访问方式枚举）

```java
public enum AccessMethod {
    SEAFILE,    // 通过 Seafile 访问
    LOCAL,      // 本地文件
    OSS         // 对象存储
}
```

**使用说明**：

| AccessMethod | 使用字段 | 说明 |
|--------------|----------|------|
| `SEAFILE` | `seafilePath` | 文件存储在 Seafile，需要通过 API 获取访问链接 |
| `LOCAL` | `filePath` | 文件存储在服务器本地文件系统 |
| `OSS` | `storageKey` | 文件存储在对象存储服务（如阿里云OSS、腾讯云COS等） |

### 2.4 业务ID生成规则

**格式**：`KB` + UUID前6位（大写）

**示例**：`KB123456`、`KBABCDEF`

**生成方法**：

```java
public static String generateFileId() {
    String uuid = UUID.randomUUID().toString().replace("-", "");
    String suffix = uuid.substring(0, 6).toUpperCase();
    return "KB" + suffix;
}
```

### 2.5 完整实体类代码

```java
package org.backend.cleanersupportagentbackend.entity;

import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Builder;
import lombok.Data;
import lombok.NoArgsConstructor;

import java.time.LocalDateTime;

/**
 * 媒体文件/知识文档实体
 */
@Entity
@Table(name = "media_files")
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class MediaFile {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true, nullable = false, length = 50)
    private String fileId; // 业务ID，如 KB001

    @Column(nullable = false, length = 200)
    private String title; // 文件标题

    @Column(nullable = false, length = 50)
    @Enumerated(EnumType.STRING)
    private FileType type; // Article, Video, PDF, Image, Excel, PPT

    @Column(length = 100)
    private String category; // 分类，如 maintenance, sales, training

    @Column(length = 500)
    private String coverUrl; // 封面图URL（可选，用于展示）

    // 存储路径字段（根据 accessMethod 决定使用哪个）
    @Column(length = 500)
    private String seafilePath; // Seafile 中的文件路径（当 accessMethod = SEAFILE 时使用）

    @Column(length = 500)
    private String filePath; // 本地文件路径（当 accessMethod = LOCAL 时使用）

    @Column(length = 500)
    private String storageKey; // 对象存储的 key（当 accessMethod = OSS 时使用）

    @Column
    private Boolean isViewable; // 是否支持在线预览（可根据文件类型自动判断）

    @Column(length = 20)
    @Enumerated(EnumType.STRING)
    private AccessMethod accessMethod; // 访问方式

    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(nullable = false)
    private LocalDateTime updatedAt;

    @PrePersist
    protected void onCreate() {
        createdAt = LocalDateTime.now();
        updatedAt = LocalDateTime.now();
    }

    @PreUpdate
    protected void onUpdate() {
        updatedAt = LocalDateTime.now();
    }

    /**
     * 文件类型枚举
     */
    public enum FileType {
        Article,   // 文章
        Video,     // 视频
        PDF,       // PDF文档
        Image,     // 图片
        Excel,     // Excel表格
        PPT        // PowerPoint演示文稿
    }

    /**
     * 访问方式枚举
     */
    public enum AccessMethod {
        SEAFILE,    // 通过 Seafile 访问
        LOCAL,      // 本地文件
        OSS         // 对象存储
    }
}
```

## 三、访问下载功能设计

### 3.1 需求概述

系统需要集成 NJU Box (Seafile) 中的媒体文件，包括：
- 销售资料（PDF、PPT、Excel）
- 产品图片和视频
- 维护视频
- 产品证书文档

这些文件存储在 NJU Box 服务器上，需要通过 API 获取访问链接，并支持：
- 在线预览（支持的文件类型）
- 文件下载
- 文件列表浏览

### 3.2 Seafile 服务封装

创建 `SeafileService` 来封装与 Seafile API 的交互：

```java
@Service
public class SeafileService {
    
    @Value("${seafile.server-url}")
    private String serverUrl;
    
    @Value("${seafile.repo-token}")
    private String repoToken;
    
    @Value("${seafile.repo-id}")
    private String repoId;
    
    /**
     * 获取文件下载链接
     * @param filePath 文件在 Seafile 中的路径
     * @return 下载链接（临时链接，有时效性）
     */
    public String getDownloadLink(String filePath);
    
    /**
     * 生成 Web 预览链接
     * @param filePath 文件在 Seafile 中的路径
     * @return 预览链接
     */
    public String generatePreviewUrl(String filePath);
    
    /**
     * 判断文件是否支持在线预览
     */
    public boolean isViewableFile(String filePath);
}
```

## 四、图片上传补充设计

### 4.1 实体关联关系

```
Message (消息实体)
  └─ recognitionId → ImageRecognition (图片识别记录)
                      └─ mediaFileId → MediaFile (媒体文件)
                                        └─ 统一管理图片访问
```

### 4.2 核心设计原则

1. **层次化关联**：
   - `Message` 只关联 `ImageRecognition`（通过 `recognitionId`）
   - `ImageRecognition` 关联 `MediaFile`（通过 `mediaFileId`）
   - 形成清晰的关联链：`Message` → `ImageRecognition` → `MediaFile`

2. **职责分离**：
   - `Message`：负责消息内容和识别记录的关联
   - `ImageRecognition`：负责图片识别信息和媒体文件的关联
   - `MediaFile`：负责统一管理图片访问（支持 LOCAL、SEAFILE、OSS）

3. **统一访问**：
   - 所有图片访问都通过 `MediaFile` 统一管理
   - 不直接存储 URL，避免 URL 变化导致的问题
   - 支持多种存储方式，易于扩展

### 4.3 数据流程

#### 图片上传流程

```
用户上传图片
  ↓
ImageRecognitionController.uploadAndRecognize
  ↓
ImageRecognitionService.recognizeImage
  ↓
保存图片到 data/images/
  ↓
创建 ImageRecognition 记录
  ↓
ImageRecognitionService.createMediaFileFromRecognition
  ↓
创建 MediaFile 记录（关联本地文件路径）
  ↓
返回 ImageRecognitionResponse（包含 imageUrl）
```

#### 发送带图片的消息流程

```
用户发送消息（包含图片）
  ↓
ChatPageNew.handleSendMessage
  ↓
获取图片信息（recognitionId）
  ↓
调用 AiService.chatWithImage
  ↓
确保 ImageRecognition 已关联 MediaFile
  ↓
保存用户消息（只包含 recognitionId）
  ↓
调用 Dify API 进行对话
  ↓
保存 AI 回复消息
  ↓
前端显示消息（通过 recognitionId -> mediaFileId 获取图片URL）
```

#### 历史会话加载流程

```
用户查看历史会话
  ↓
调用 getConversationDetail API
  ↓
AiService.getConversationDetail
  ↓
查询消息列表（包含 recognitionId 字段）
  ↓
通过 recognitionId 查询 ImageRecognition
  ↓
通过 ImageRecognition.mediaFileId 获取 MediaFile
  ↓
通过 MediaService 获取每个消息的图片URL
  ↓
构建 MessageResponse（包含 recognitionId、mediaFileId 和 imageUrl）
  ↓
前端接收消息列表
  ↓
渲染消息（显示图片，优先使用 imageUrl，如果没有则通过 mediaFileId 获取）
```

## 五、实体类重构说明

### 5.1 重构原因

原设计中，`MediaFile` 实体类直接存储了以下字段：
- `previewUrl`：预览链接
- `downloadUrl`：下载链接
- `mediaUrl`：媒体文件URL
- `repoId`：Seafile 资料库ID

**问题**：
1. **临时链接有时效性**：`previewUrl` 和 `downloadUrl` 是临时链接，存储后很快就会失效
2. **数据冗余**：`repoId` 是配置信息，不需要每个文件都存储
3. **URL 可生成**：`mediaUrl` 可以通过文件路径生成，不需要存储
4. **违反设计原则**：应该存储路径信息，通过服务生成访问链接

### 5.2 重构内容

#### 移除的字段

| 字段名 | 原因 | 替代方案 |
|--------|------|----------|
| `previewUrl` | 临时链接，有时效性 | 通过 `SeafileService.generatePreviewUrl()` 实时生成 |
| `downloadUrl` | 临时链接，有时效性 | 通过 `SeafileService.getDownloadLink()` 实时生成 |
| `mediaUrl` | 可通过路径生成 | 通过 `MediaService.generateMediaUrl()` 实时生成 |
| `repoId` | 配置信息，不需要每个文件存储 | 从 `application.yml` 配置中读取 |

#### 新增的字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `storageKey` | `String(500)` | 对象存储的 key（当 `accessMethod = OSS` 时使用） |

### 5.3 服务层改动

#### MediaService 新增方法

1. **`generateMediaUrl(MediaFile file)`**
   - 根据访问方式生成媒体URL
   - Seafile：返回预览链接
   - LOCAL：生成本地文件访问URL
   - OSS：生成OSS访问URL

2. **`getFileDownloadLink(String fileId)`**
   - 根据访问方式实时生成下载链接
   - Seafile：调用 `SeafileService.getDownloadLink()`
   - LOCAL：调用 `generateLocalFileUrl()`
   - OSS：调用 `generateOssFileUrl()`

3. **`getFilePreviewUrl(String fileId)`**
   - 根据访问方式实时生成预览链接
   - Seafile：调用 `SeafileService.generatePreviewUrl()`
   - LOCAL：调用 `generateLocalFileUrl()`
   - OSS：调用 `generateOssFileUrl()`

### 5.4 优势

1. **数据一致性**：不存储临时链接，避免链接失效问题
2. **减少存储**：移除冗余字段，减少数据库存储空间
3. **灵活性**：链接生成逻辑集中管理，易于修改和维护
4. **安全性**：临时链接实时生成，提高安全性
5. **配置集中**：`repoId` 等配置信息集中管理

### 6.4 常见问题

#### Q1: 如何获取文件的 fileId？

**答**：通过文件列表接口获取。列表接口返回的每项都包含 `id` 字段，这就是 `fileId`。

#### Q2: 文件预览失败怎么办？

**可能原因**：
1. 文件类型不支持预览（如 Excel、PPT）
2. Seafile 服务器连接失败
3. 文件路径不存在

**解决方法**：
1. 检查 `isViewable` 字段，确认文件是否支持预览
2. 检查 Seafile 配置是否正确
3. 查看后端日志，确认具体错误信息

#### Q3: 下载链接失效怎么办？

**说明**：Seafile 的下载链接是临时链接，有时效性。如果链接失效，需要重新调用接口获取新的下载链接。

**解决方法**：每次下载前都调用 `/files/{id}/download` 接口，让后端实时获取新的下载链接。

#### Q4: 如何判断文件是否支持预览？

**方法1**：通过访问信息接口

```typescript
const response = await fetch(`/api/cleaner-support/v2/media/files/${fileId}/access`);
const result = await response.json();
const isViewable = result.data.isViewable;
```

**方法2**：通过文件详情接口

```typescript
const response = await fetch(`/api/cleaner-support/v2/media/files/${fileId}`);
const result = await response.json();
const isViewable = result.data.isViewable;
```

### 6.5 最佳实践

#### 6.5.1 前端建议

1. **列表查询**：使用分页，避免一次性加载大量数据
2. **预览文件**：在新窗口打开，避免影响当前页面
3. **下载文件**：显示下载进度（如果可能）
4. **错误处理**：友好提示用户错误信息
5. **缓存策略**：文件列表可以适当缓存，但访问链接不要缓存（因为有时效性）

#### 6.5.2 后端建议

1. **配置安全**：生产环境使用环境变量，不要硬编码敏感信息
2. **错误处理**：记录详细日志，便于排查问题
3. **性能优化**：为常用查询字段添加索引
4. **链接缓存**：可以考虑缓存下载链接（但要注意时效性）

## 七、数据库表结构

### 7.1 主表结构（media_files）

```sql
CREATE TABLE media_files (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    file_id VARCHAR(50) UNIQUE NOT NULL,
    title VARCHAR(200) NOT NULL,
    type VARCHAR(50) NOT NULL,
    category VARCHAR(100),
    cover_url VARCHAR(500),
    -- 存储路径字段（根据 access_method 决定使用哪个）
    seafile_path VARCHAR(500),
    file_path VARCHAR(500),
    storage_key VARCHAR(500),
    is_viewable BOOLEAN,
    access_method VARCHAR(20),
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL,
    
    INDEX idx_file_id (file_id),
    INDEX idx_category (category),
    INDEX idx_type (type),
    INDEX idx_access_method (access_method),
    INDEX idx_created_at (created_at DESC)
);
```

**注意**：以下字段**不应存储在数据库中**：
- `preview_url`：预览链接（临时链接，通过服务实时生成）
- `download_url`：下载链接（临时链接，通过服务实时生成）
- `media_url`：媒体文件URL（通过路径生成）
- `repo_id`：Seafile 资料库ID（从配置中读取）
