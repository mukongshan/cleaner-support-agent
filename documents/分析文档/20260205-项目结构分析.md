### 项目总览

- **项目名**: `cleaner-support-agent-backend`
- **技术栈**: Spring Boot 3.2、Java 21、Spring Web / WebFlux、Spring Data JPA、MySQL（附带 H2 测试）、JWT、Lombok
- **模块定位**: 为「清洁助手」应用提供后台服务，包含账号与鉴权、会话与对话消息、媒体文件管理、图片识别、工单流转、与外部 AI 服务（Dify、通义千问多模态）以及 Seafile 文件存储的集成。

项目采用典型的单体 Spring Boot 分层结构：`controller`（接口层） → `service`（业务层） → `repository`（数据访问层） → `entity`（领域模型），并通过 `dto` 进行请求/响应数据的封装，`config` 负责全局配置与基础设施，`util` 提供通用工具函数。

### Maven 与运行入口

- **构建工具**: Maven（`pom.xml`）
  - 继承 `spring-boot-starter-parent:3.2.0`，约定整体依赖与插件版本。
  - 关键依赖：
    - **Web 层**: `spring-boot-starter-web`（传统 MVC）、`spring-boot-starter-webflux`（主要用于 REST Client）
    - **数据访问**: `spring-boot-starter-data-jpa` + `mysql-connector-j` + `h2`（测试/本地使用）
    - **安全与鉴权**: `spring-security-crypto`（密码加密）、`jjwt-*`（JWT 生成与解析）
    - **开发辅助**: `lombok`
    - **测试**: `spring-boot-starter-test`
  - 编译插件：
    - `maven-compiler-plugin`（配置 Lombok 注解处理）
    - `spring-boot-maven-plugin`（打包/运行 Spring Boot 应用）

- **启动类**: `CleanerSupportAgentBackendApplication`
  - 包路径：`org.backend.cleanersupportagentbackend`
  - 作用：标准 Spring Boot 启动入口，负责启动整个单体后端服务。

### 代码包结构总览（`src/main/java/org/backend/cleanersupportagentbackend`）

#### 1. `annotation`
- **`CurrentUserId`**
  - 自定义注解，用于在控制层方法参数中注入当前登录用户 ID（通常配合 `HandlerMethodArgumentResolver` 使用）。

#### 2. `config`
- **`AuthInterceptor`**
  - 基于 JWT 的请求拦截器，负责从请求头解析 Token，校验合法性与过期时间，将用户信息放入上下文，供后续处理链使用。
- **`CorsConfig`**
  - 全局 CORS 配置，通常从 `app.cors.allowed-origins` 读取前端允许来源，解决前后端分离的跨域问题。
- **`CurrentUserIdArgumentResolver`**
  - 自定义参数解析器，配合 `@CurrentUserId` 注解，将当前用户 ID 注入到控制器方法参数中，简化获取登录用户的代码。
- **`DifyConfig`**
  - 负责读取 `app.dify.*` 相关配置（如 `base-url`、`api-key`、`user-prefix` 等），为 `DifyClient` 提供配置来源。
- **`GlobalExceptionHandler`**
  - 全局异常处理类，通过 `@ControllerAdvice` 统一捕获后端抛出的业务异常、校验异常、系统异常，并转换为统一的 `ApiResponse` 格式返回前端。
- **`PasswordEncoderConfig`**
  - 配置密码加密器（如 BCryptPasswordEncoder），应用在 `UserService` 等涉及密码存储与校验的地方。
- **`WebConfig`**
  - Spring MVC 相关扩展配置：
    - 注册 `AuthInterceptor`、`CurrentUserIdArgumentResolver`
    - 注册全局 CORS 配置
    - 统一配置静态资源、消息转换等（视项目需求而定）

#### 3. `controller`

- **`ApiResponse`**
  - 统一响应封装类，一般包含 `code`、`message`、`data` 等字段，为所有接口输出提供统一结构。

- **`controller.app`**（对前端 / App 暴露的业务接口）
  - **`AiController`**
    - 会话聊天相关接口：
      - 发送文本或图文消息到 AI 服务（Dify / Qwen-VL）
      - 获取会话列表与摘要
      - 查询历史消息记录
      - 提交消息反馈（有助于效果评估）
    - 背后依赖：`AiService`、`ConversationRepository`、`MessageRepository`、`MessageFeedbackRepository` 等。
  - **`ImageRecognitionController`**
    - 图片识别相关接口：
      - 上传图片/传入 Base64 触发识别
      - 查询识别结果与历史记录
    - 背后依赖：`ImageRecognitionService`、`QwenVLClient`、`ImageRecognitionRepository`。
  - **`MediaController`**
    - 媒体文件（图片/文档等）管理：
      - 文件上传、持久化元数据
      - 列表查询与详情
      - 下载与重定向（配合 Seafile 或本地文件系统）
    - 背后依赖：`MediaService`、`MediaFileRepository`、`SeafileService`。
  - **`TicketController`**
    - 工单生命周期管理：
      - 创建工单、更新工单状态
      - 查询工单详情 / 列表
      - 工单反馈记录
    - 背后依赖：`TicketService`、`TicketRepository`、`TicketFeedbackRepository`。
  - **`UserController`**
    - 用户与账号相关接口：
      - 注册 / 登录（JWT 签发）
      - 获取 / 更新个人资料
    - 背后依赖：`UserService`、`UserRepository`、`TokenUtil`、`PasswordEncoderConfig`。

- **`controller.mcp`**
  - **`McpController`**
    - 面向 MCP（可能是 agent / 工具调用）的接口适配层，将内部能力以 MCP 规范/协议暴露外部，便于 agent 自动化调用。

#### 4. `dto`（数据传输对象）

- **会话与聊天**
  - `ChatRequest` / `ChatWithImageRequest`：前端提交的聊天请求（纯文本或附带图片）。
  - `ConversationDetailResponse` / `ConversationSummaryResponse`：会话详情与摘要信息。
  - `MessageResponse`：单条消息的返回结构。
  - `DifyEvent`：接收或发送与 Dify 平台交互的事件数据结构。

- **认证与用户**
  - `LoginRequest` / `RegisterRequest`：登录、注册请求体。
  - `LoginResponse`：登录成功后返回的 JWT 等信息。
  - `UserProfileResponse` / `UpdateUserProfileRequest`：用户个人资料的查询与更新。

- **媒体文件与图片识别**
  - `UploadFileResponse`：文件上传后的返回信息（如文件 ID、原始文件名、URL 等）。
  - `MediaFileSummaryResponse` / `MediaFileDetailResponse`：媒体文件列表项与详情项结构。
  - `ImageRecognitionBase64Request`：使用 Base64 提交的图片识别请求。
  - `ImageRecognitionResponse`：图片识别结果返回体。
  - `ImageRecognitionHistoryResponse`：历史识别记录列表项。
  - `FileAccessInfo` / `FileRedirectInfo`：与 Seafile 或本地文件 URL/重定向有关的信息封装。

- **工单系统**
  - `CreateTicketRequest` / `UpdateTicketRequest`：创建与更新工单请求体。
  - `TicketDetailResponse` / `TicketSummaryResponse`：工单详情与列表摘要。

#### 5. `entity`（实体 / 领域模型）

- **`User`**
  - 用户账号实体，字段一般包括：用户名、邮箱、加密后的密码、角色/权限、创建更新时间等。

- **`Conversation`、`Message`、`MessageFeedback`**
  - 用于记录与 AI 的对话会话及消息明细，以及用户对单条消息的反馈。

- **`MediaFile`**
  - 媒体文件元数据实体，包括：文件名、类型、大小、存储位置（本地路径/Seafile 信息）、关联业务（如工单、知识文档等）。

- **`ImageRecognition`、`RecognitionStatus`**
  - 图片识别任务及状态：
    - `ImageRecognition`：一次识别任务的输入（图片）与输出（识别结果、耗时等）。
    - `RecognitionStatus`：识别任务状态（如 PENDING、PROCESSING、SUCCESS、FAILED）。

- **`Ticket`、`TicketFeedback`**
  - 工单主体与工单反馈记录：
    - `Ticket`：描述问题、状态、优先级、关联用户等。
    - `TicketFeedback`：用户或客服对工单的评价反馈。

#### 6. `repository`（数据访问层）

- 使用 Spring Data JPA 定义的 Repository 接口：
  - `UserRepository`
  - `ConversationRepository`
  - `MessageRepository`
  - `MessageFeedbackRepository`
  - `MediaFileRepository`
  - `ImageRecognitionRepository`
  - `TicketRepository`
  - `TicketFeedbackRepository`
- 主要职责：
  - 通过方法名或自定义查询访问关系型数据库中的对应实体表。
  - 由 Spring 自动生成实现，service 直接注入使用。

#### 7. `service`（业务逻辑层）

- **核心业务服务**
  - `AiService`
    - 负责与外部 AI 服务（主要是 Dify）交互：
      - 组装聊天请求（上下文、用户标识、会话信息）。
      - 将 Dify 返回的结果持久化为 `Conversation`、`Message`、`MessageFeedback` 等实体。
      - 封装给上层 controller 使用的业务接口。
  - `MediaService`
    - 处理媒体文件相关业务逻辑：
      - 文件元数据保存与查询。
      - 结合 `SeafileService` 或本地文件系统，生成下载地址或访问重定向信息。
  - `TicketService`
    - 工单全生命周期管理：
      - 创建、更新、关闭工单。
      - 关联用户、媒体文件、对话记录等信息。
  - `UserService`
    - 账号与用户信息维护：
      - 注册（密码加密）、登录（校验密码并生成 JWT）、用户资料修改。
      - 结合 `TokenUtil`、`PasswordEncoderConfig`、`UserRepository` 实现完整登录态管理。

- **client 子包**
  - `DifyClient`
    - 封装与 Dify 平台 HTTP 调用的底层细节：
      - 使用 `DifyConfig` 提供的 `base-url`、`api-key` 等配置。
      - 对 WebClient / RestTemplate 请求进行统一封装，供 `AiService` 调用。
  - `QwenVLClient`
    - 封装与通义千问多模态（Qwen-VL）服务的 HTTP 调用：
      - 使用 `app.image-recognition.qwen-vl.*` 配置（API URL、API Key、模型、超时等）。
      - 对图片识别场景进行特别封装。

- **support 子包**
  - `ImageRecognitionService`
    - 面向图片识别业务的组合服务：
      - 将上传/传入的图片交给 `QwenVLClient` 处理。
      - 将识别任务与结果以 `ImageRecognition` 实体形式持久化。
      - 提供识别历史查询接口给上层 `ImageRecognitionController`。
  - `SeafileService`
    - 封装 Seafile 存储相关逻辑：
      - 使用 `seafile.*` 配置（`server-url`、`repo-id`、`repo-token` 等）。
      - 提供生成下载链接、上传文件、权限控制等能力。

#### 8. `util`

- **`IdGenerator`**
  - 通用 ID 生成工具，用于为实体或业务对象生成唯一标识（例如基于时间戳/随机数/雪花算法等）。
- **`TokenUtil`**
  - JWT 工具类：
    - 使用 `app.jwt.secret` 与 `app.jwt.expiration` 配置。
    - 提供生成、解析、校验 Token 的静态/实例方法。

### 资源与配置结构（`src/main/resources`）

- **`application.yml`**
  - 核心配置汇总：
    - `spring.application.name`: 服务名称。
    - `spring.profiles.active`: 默认 `dev`，可通过环境变量覆盖。
    - `spring.profiles.include: local`: 引入本地配置（仅本机，不提交仓库）。
    - `spring.datasource.*`: 数据源配置占位，鼓励使用环境变量或本地文件注入。
    - `spring.jpa.*`: JPA 相关配置（DDL 策略、SQL 日志等，可通过环境变量覆写）。
    - `spring.servlet.multipart.*`: 上传文件大小限制。
    - `server.*`: 端口、优雅停机、错误信息展示等。
    - `logging.*`: 日志级别与控制台输出格式。
    - `management.*`: Actuator 端点暴露与健康检查详略。
    - `app.cors.*`: CORS 允许的前端来源。
    - `app.dify.*`: Dify 平台访问配置。
    - `app.media.*`: 媒体文件上传目录等本地存储配置。
    - `app.image-recognition.*`: Qwen-VL 相关参数与图片上传约束。
    - `app.jwt.*`: JWT 密钥与过期时间。
    - `seafile.*`: Seafile 服务器与仓库访问配置。
  - 多 profile 支持：
    - `---` + `spring.config.activate.on-profile=dev`：开发环境日志更详细（web 层为 DEBUG）。
    - `---` + `spring.config.activate.on-profile=prod`：生产环境的基础占位配置。

- **`application-local.yml`**
  - 被 `.gitignore` 忽略，仅用于本机开发，存放敏感信息（数据库密码、Dify/Qwen-VL/Seafile 的密钥和地址等）。
  - 通过 `spring.profiles.include=local` 自动加载，但不会影响 CI / 生产环境。

- **`db` 目录**
  - `init.sql` / `init_media_files.sql`：
    - 提供数据库初始化脚本（建表、基础数据），用于本地开发或测试环境快速初始化。
  - `README.md`：
    - 说明如何使用这些 SQL 脚本、依赖的数据库环境等。

### 小结：整体架构特点

- **分层清晰、职责明确**：controller / service / repository / entity / dto / config 各司其职，便于扩展与维护。
- **对外能力丰富**：支持用户管理、会话聊天、媒体文件管理、图片识别、工单管理以及面向 MCP 的接口。
- **良好的可配置性**：针对数据库、AI 服务（Dify、Qwen-VL）、文件存储（Seafile）、JWT、CORS 等均通过配置文件与环境变量集中管理。
- **测试保障**：为主要控制器提供了单元/集成测试，有利于后续重构与新功能迭代。
