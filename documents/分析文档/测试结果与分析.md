- **运行命令**：`mvnw.cmd test`

- **测试类结构**：
  - `BaseControllerTest`（公共基类，封装登录逻辑）
  - `AuthControllerTest`
  - `AiControllerTest`

  - `MediaControllerTest`
  - `CleanerSupportAgentBackendApplicationTests`（context load）

## 二、各测试类执行结果简要

### 2.1 AuthControllerTest

- **通过**：
  - `login_shouldReturnToken`：`POST /users/login` 能返回 `code=200` 和 mock token。
  - `profile_shouldBeUnauthorizedWithoutToken`：未带 token 访问 `GET /users/profile`，返回 `code=401`。
- **失败**：
  - `profile_getAndUpdate_shouldWorkWithToken`：带 token 访问 `GET /users/profile` 期望 `code=200`，实际 `code=500`、消息 `"用户不存在"`。
- **原因**：登录时虽创建了用户，但当前 `UserService.getUserByUserId` / 相关逻辑仍不稳定，在 Profile 查询时抛出了 `"用户不存在"` 异常，被全局异常处理为 `code=500`。

### 2.2 AiControllerTest

- **通过**：
  - `conversations_shouldRequireAuth`：未带 token 访问 `GET /ai/conversations`，返回 `code=401`。
- **失败**：
  - `chat_shouldReturnEventStream`：期望 `/ai/chat` 返回 `text/event-stream`，实际返回 JSON `code=500`（`"用户不存在"`）。
  - `conversations_listAndDetail_shouldWorkAfterChat`：调用 `/ai/chat` 后访问会话列表与详情，期望 `code=200`，实际仍是 `code=500`、`"用户不存在"`。
- **原因**：
  - `AiService.chat` 仍是占位实现，内部依赖 `UserService.getUserByUserId`，同样抛出 `"用户不存在"`。
  - SSE 逻辑尚未接好，返回的是统一错误 JSON，而不是事件流。

### 2.3 TicketControllerTest

- **失败**：
  - `tickets_crud_shouldWorkWithAuth`：创建工单 `POST /tickets` 期望 `code=200`，实际 `code=500`、`"用户不存在"`；后续列表 / 详情 / 更新自然也无法继续验证。
- **原因**：`TicketService.createTicket` 通过 `UserService.getUserByUserId` 获取用户时失败，属于同一类“用户不存在”问题，业务未完全实现。

### 2.4 MediaControllerTest

- **通过**：
  - `listFiles_shouldBePublic`：`GET /media/files` 不带 token，`code=200`，`data` 为数组（目前为空）。
  - `fileDetail_shouldReturnErrorForUnknownId`：`GET /media/files/KB_NOT_EXISTS` 返回 `code=500`、`"文件不存在"`，与当前实现一致。
  - `upload_shouldRequireAuth_andWorkWithAuth`：
    - 不带 token 上传 → `code=401`；
    - 带 token 上传 → `code=200`，`data.fileType="image"`，URL 合法。
- **说明**：媒体上传与访问链路（含认证拦截器）工作正常。

### 2.5 CleanerSupportAgentBackendApplicationTests

- **通过**：
  - `contextLoads`：应用上下文能够正常启动，无 Bean 装配错误。

## 三、总体结论（结构化总结）

### 3.1 基础设施与连通性

- **应用与数据库连通性良好**：
  - Spring Boot 能在本地 MySQL 环境下正常启动。
  - JPA 能根据实体自动创建 / 更新表结构。
- **HTTP 路由与拦截链正常**：
  - 所有 v2 控制器的 URL 映射正常工作（无 404）。
  - 认证拦截器可以正确区分“需登录接口”和“公开接口”。
  - CORS 过滤器已生效，适配前端跨域访问。

**结论 1**：从“系统基础能力”和“接口连通性”角度看，整体架构是健康可用的，适合继续在其上迭代业务逻辑。

### 3.2 已经基本可用的模块

- **媒体模块（Media）**：
  - 列表查询、错误场景（文件不存在）、带认证的文件上传，均能按照当前实现逻辑稳定通过测试。
  - 说明上传链路（Multipart 处理、本地存储、返回 URL）已经可以用于前端联调。
- **认证基础链路**：
  - 登录接口能够返回可用的 mock token。
  - 依赖 Authorization Header 的拦截逻辑在各控制器中表现一致。

**结论 2**：媒体相关功能已经具备“可 Demo / 可联调”的状态；统一认证链路也已经打通，只差将 mock token 替换为正式 JWT 实现。

### 3.3 当前主要的业务缺口

- **用户实体的使用尚不统一**：
  - 多个 Service（AiService、TicketService、UserService 本身）依赖 `UserService.getUserByUserId`，当前仍可能抛出 `"用户不存在"`，导致接口返回 `code=500`。
  - 登录流程中创建的用户，与随后按业务 ID 查询的用户之间，生命周期和一致性需要梳理（例如：是否允许“自动注册”、如何区分测试用户和真实用户）。
- **AI 对话业务未实现到位**：
  - `/ai/chat` 仍停留在抛 RuntimeException 的阶段，没有真正返回 SSE 事件流，也没有与 Dify / 会话记录完全打通。
  - 会话列表和详情接口虽然有测试，但依赖于完整的“对话存储逻辑”，目前尚未落地，因此返回 500。
- **工单业务流程尚未打通**：
  - 工单创建 / 查询 / 状态更新的接口已经定义并有测试，但由于依赖用户信息，同样受 `"用户不存在"` 问题影响，暂时无法跑通完整 happy path。

**结论 3**：失败的测试集中暴露了“用户加载、AI 对话、工单流程”三条业务主线尚未完善，而非框架 / 基础设施问题。

### 3.4 对当前阶段测试结果的解读

- 现有控制器测试**已经起到了“连通性 + 业务期望”的双重作用**：
  - 连通性层面（路由、拦截器、序列化）已经验证通过；
  - 业务层面（`code=200` + 具体数据结构）在尚未实现完逻辑时自然会失败。
- 在当前阶段，可以将这些失败测试看作是**业务开发的待办清单**：
  - 当某个接口实现完成后，对应的测试用例就会从红变绿，自动起到“回归测试”的作用。

**结论 4**：从工程实践角度，当前的测试状态是“基础架构稳定、业务逻辑待完善”的合理中间态，后续开发可以以这些测试为目标逐步补齐实现。

## 四、后续工作建议（按优先级）

### 4.1 优先级一：修复“用户不存在”链路

1. 统一梳理 `UserService.getUserByUserId` 的使用场景：
   - 登录时是否自动创建用户？
   - 如果 token 中 userId 不存在，返回 401 还是 404？
2. 在测试环境中，约定一套稳定的“测试用户”策略：
   - 例如：登录时必定创建用户，后续调用不再抛出 `"用户不存在"`。
3. 修复后，Auth / Ai / Ticket 相关测试会大量从 `500` 变为 `200`。

### 4.2 优先级二：打通 AI 对话最小闭环

1. 为 `/ai/chat` 实现最小可用逻辑：
   - 即使暂时不接 Dify，也可以返回一个固定 SSE 文本，并在数据库中创建会话与消息记录。
2. 确保会话列表 / 详情接口能基于真实数据返回结果。
3. 修正 `AiControllerTest` 里对 `code=200` 与 SSE 的断言。

### 4.3 优先级三：工单 CRUD 全流程

1. 使用已经存在的实体（`Ticket`）和 Service（`TicketService`）补齐：
   - 创建 → 列表 → 详情 → 更新状态 的完整链路。
2. 根据 API 文档 v2.0 细化返回字段和状态流转规则。
3. 让 `TicketControllerTest` 变为全部通过。

## 五、小结

- **从“系统连通性”和“基础设施”角度**，当前后端已经可以稳定启动并连接 MySQL，各主要控制器的路由和拦截链路正常工作。
- **从“业务完备度”角度**，用户加载、AI 对话和工单管理仍处于“有骨架、未填肉”的阶段，测试已经准确地暴露出这些空白点。
- 随着业务逻辑逐步补齐，当前这些测试文件可以直接充当“回归测试集”，帮助你判断每次改动是否破坏了既有行为。
